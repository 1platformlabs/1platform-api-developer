# Environments

The 1Platform API is deployed to two isolated environments. API keys are scoped
to a single environment — a QA key returns `401` against production and vice
versa.

| Environment | Base URL | OpenAPI |
|---|---|---|
| Production | `https://api.1platform.pro/api/v1` | [openapi.json](https://api.1platform.pro/openapi.json) |
| QA / sandbox | `https://api-qa.1platform.pro/api/v1` | [openapi.json](https://api-qa.1platform.pro/openapi.json) |

## Choosing an environment

- **QA** — integration testing, sandbox payment gateway, throwaway tenants.
- **Production** — live tenants, real payments. Provisioned only after the
  integration passes acceptance.

To request QA credentials, contact your account manager. Both QA and prod keys
carry the same `ak-` (app) / `sk-` (user) prefixes — the only difference is
the host they authenticate against.

:::warning Don't mix environments in one codebase
Treat the base URL and the API keys as a single bundle. The simplest pattern
is two distinct `.env` files (`.env.qa` / `.env.prod`) plus a single
`API_BASE_URL` variable that your client constructor reads at startup.
:::

## What differs between QA and prod

| Concern | QA | Production |
|---|---|---|
| Base URL | `api-qa.1platform.pro` | `api.1platform.pro` |
| Rate limits | Identical | Identical |
| Payment gateway | Sandbox only — no real money flows | Real gateway |
| Test card numbers | See [Testing](./testing) | N/A |
| Tenant data | Isolated; nothing migrates between environments | Live data |
| Webhook signing | Per-domain `webhook_secret` | Per-domain `webhook_secret` |

## Common pitfalls

- **`401` against prod with QA keys** — the most common cause of "my key
  is valid but I can't authenticate". Inspect the `Host:` header your client
  is sending.
- **Old key formats** — keys minted before 2026-04-29 are raw 64-char hex.
  Newer keys carry `sk-` / `ak-` prefixes. Both work; the API does an
  exact-match lookup against `User.api_key` / `App.api_key`.

## Programmatic discovery

To distinguish environments at runtime, hit `GET /health` and inspect the
returned `environment` field:

```bash
curl -s https://api-qa.1platform.pro/api/v1/health | jq '.data.environment'
# "qa"
```
