# Flujo: Gestión de dominios

## Vista general

Este flujo documenta cómo gestionar dominios a través de la 1Platform API. Cubre la verificación de disponibilidad de dominios, la obtención de precios por TLD, el registro de dominios, la gestión de registros DNS y nameservers, el inicio de transferencias, la renovación de dominios y el seguimiento del historial de operaciones.

**Caso de uso:** Brinda a tus usuarios una experiencia completa de gestión de dominios -- desde la búsqueda y el registro de dominios hasta la configuración de registros DNS y el manejo de transferencias -- todo a través de una única API.

**Resultado final:** Un dominio totalmente registrado con registros DNS configurados, privacidad WHOIS y la capacidad de gestionar nameservers, renovaciones, bloqueos y transferencias de forma programática.

---

## Requisitos previos

| Requisito | Detalles |
|---|---|
| App API Key | Clave de aplicación (`ak-...`) provista al registrar tu app |
| User API Key | Clave de usuario (`sk-...`) obtenida al crear el usuario |
| Saldo suficiente | El registro y la renovación de dominios requieren saldo en la cuenta del usuario |
| Contacto de dominio | Debe configurarse un perfil de contacto antes de registrar dominios |

### Headers requeridos para todas las solicitudes

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

## Variables usadas en los ejemplos

```bash
BASE_URL="https://api.1platform.pro/api/v1"
APP_API_KEY="ak-your-app-api-key"
USER_API_KEY="sk-user-abc123"
APP_TOKEN=""      # Obtenido en el Paso 0
USER_TOKEN=""     # Obtenido en el Paso 1
DOMAIN_ID=""      # Obtenido tras el registro del dominio (Paso 5)
```

## Datos requeridos

### Contacto

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `first_name` | string | Sí | Nombre del contacto (máx. 100 caracteres) |
| `last_name` | string | Sí | Apellido del contacto (máx. 100 caracteres) |
| `email` | string | Sí | Email del contacto (máx. 255 caracteres, formato email) |
| `phone` | string | Sí | Teléfono en formato E.164 con punto: `+N.NNNNNNN` (máx. 20 caracteres) |
| `address` | string | Sí | Dirección (máx. 255 caracteres) |
| `city` | string | Sí | Ciudad (máx. 100 caracteres) |
| `state` | string | Sí | Estado o provincia (máx. 100 caracteres) |
| `country` | string | Sí | Código de país ISO 3166-1 alpha-2 (exactamente 2 caracteres) |
| `postal_code` | string | Sí | Código postal/ZIP (máx. 20 caracteres) |
| `organization` | string o null | No | Nombre de la organización (máx. 255 caracteres) |

### Registro

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `domain` | string | Sí | Nombre de dominio a registrar, p. ej. `"example.com"` (máx. 253 caracteres) |
| `years` | integer | No | Período de registro, 1-10 (por defecto: 1) |
| `whois_privacy` | boolean | No | Habilitar privacidad WHOIS (por defecto: true) |
| `auto_renew` | boolean | No | Habilitar renovación automática (por defecto: false) |
| `nameservers` | array de strings o null | No | Nameservers personalizados (máx. 12) |

### Registros DNS

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `type` | string | Sí | Tipo de registro: `A`, `AAAA`, `CNAME`, `MX`, `TXT`, `NS`, `URL_REDIRECT`, `URL_FRAME` |
| `host` | string | Sí | Hostname, p. ej. `"@"`, `"www"`, `"mail"` (máx. 253 caracteres) |
| `value` | string | Sí | Valor del registro: IP, hostname, texto, etc. (máx. 1024 caracteres) |
| `ttl` | integer | No | Tiempo de vida en segundos, 60-86400 (por defecto: 1800) |
| `mx_pref` | integer o null | No | Prioridad MX, 0-65535 (requerido para registros MX) |

### Transferencia

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `domain` | string | Sí | Dominio a transferir (máx. 253 caracteres) |
| `epp_code` | string | Sí | Código EPP/auth del registrador actual (máx. 255 caracteres) |
| `years` | integer | No | Años a añadir en la transferencia, 1-10 (por defecto: 1) |
| `whois_privacy` | boolean | No | Habilitar privacidad WHOIS (por defecto: true) |
| `auto_renew` | boolean | No | Habilitar renovación automática (por defecto: false) |

## Flujo paso a paso

### Paso 0 — Obtener el token de aplicación

**Objetivo:** Autenticar la aplicación y obtener el token JWT de la app.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** Ninguna (este es el punto de entrada)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$APP_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key inválida o app inactiva | Verifica que `apiKey` sea correcta y que la app esté activa |
| 422 | Campo `apiKey` ausente o malformado | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido (10 req/min) | Espera y reintenta |

### Paso 1 — Obtener el token de usuario

**Objetivo:** Autenticar al usuario y obtener el token JWT del usuario.

**Endpoint:** `POST /api/v1/users/token`

**Auth:** `Authorization: Bearer $APP_TOKEN` (solo token de aplicación, no se requiere token de usuario)

```bash
curl -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$USER_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "User token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$USER_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key de usuario inválida o usuario inactivo | Verifica el `apiKey` del usuario |
| 422 | Campo `apiKey` ausente | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido | Espera y reintenta |

:::tip
Si aún no tienes un usuario, créalo con `POST /api/v1/users` enviando `{"email": "user@example.com"}`. La respuesta incluirá el `api_key` del usuario.
:::

### Paso 2 — Crear o actualizar el contacto de dominio

**Objetivo:** Configurar el perfil de contacto usado para el registro de dominios. Esto debe hacerse antes de registrar cualquier dominio.

**Endpoint:** `PUT /api/v1/domains/contact`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X PUT "$BASE_URL/domains/contact" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "first_name": "Carlos",
    "last_name": "Ejemplo Demo",
    "email": "carlos@example.com",
    "phone": "+502.00001234",
    "address": "Avenida Reforma 10-50",
    "city": "Guatemala",
    "state": "Guatemala",
    "country": "GT",
    "postal_code": "01010",
    "organization": null
  }'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "first_name": "Carlos",
    "last_name": "Ejemplo Demo",
    "email": "carlos@example.com",
    "phone": "+502.00001234",
    "address": "Avenida Reforma 10-50",
    "city": "Guatemala",
    "state": "Guatemala",
    "country": "GT",
    "postal_code": "01010",
    "organization": null,
    "updated_at": "2026-03-18T10:00:00"
  },
  "msg": "Contact saved successfully"
}
```

:::note
Este endpoint es idempotente -- llamarlo varias veces actualiza el mismo perfil de contacto. Solo necesitas un contacto por usuario.
:::

:::warning
Se **requiere** un perfil de contacto antes de registrar cualquier dominio (Paso 5) o iniciar una transferencia (Paso 13). Si el contacto no está configurado, esos endpoints devuelven un error **404**.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 400 | Datos de contacto inválidos (validación fallida) | Verifica todos los campos y formatos requeridos (p. ej., `country` debe tener 2 caracteres, `phone` debe coincidir con `+N.NNNNNNN`) |
| 401 | Se requiere autenticación | Regenera los tokens |
| 422 | Campos ausentes o malformados | Verifica que todos los campos requeridos estén presentes y sean válidos |
| 429 | Límite de tasa excedido | Espera y reintenta |

### Paso 3 — Verificar disponibilidad de dominio

**Objetivo:** Verificar si uno o más dominios están disponibles para registro.

**Endpoint:** `GET /api/v1/domains/check`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X GET "$BASE_URL/domains/check?domain=example-demo.com&domain=tienda-ejemplo.com" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": [
    {
      "domain": "example-demo.com",
      "available": true,
      "price": 12.99,
      "currency": "USD",
      "period": 1
    },
    {
      "domain": "tienda-ejemplo.com",
      "available": false,
      "price": null,
      "currency": null,
      "period": null
    }
  ],
  "msg": "Domain availability checked"
}
```

:::tip
Puedes verificar hasta **50 dominios** en una sola solicitud repitiendo el parámetro de consulta `domain`.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 400 | Formato de dominio inválido o demasiados dominios (máx. 50) | Verifica el formato del dominio y reduce la cantidad de dominios |
| 401 | Se requiere autenticación | Regenera los tokens |
| 422 | Parámetro `domain` ausente o malformado | Proporciona al menos un parámetro de consulta `domain` |
| 429 | Límite de tasa excedido | Espera y reintenta |
| 502 | Proveedor de dominios temporalmente no disponible | Reintenta después de unos segundos |
| 503 | Servicio de dominios deshabilitado | Contacta a soporte -- el servicio de dominios puede estar en mantenimiento |

### Paso 4 — Obtener precios por TLD

**Objetivo:** Obtener los precios de registro para uno o más TLDs.

**Endpoint:** `GET /api/v1/domains/pricing`

```bash
curl -X GET "$BASE_URL/domains/pricing?tld=com&tld=io&tld=dev" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": [
    {
      "tld": "com",
      "register_price": 12.99,
      "renew_price": 12.99,
      "transfer_price": 9.99,
      "currency": "USD"
    },
    {
      "tld": "io",
      "register_price": 39.99,
      "renew_price": 39.99,
      "transfer_price": 39.99,
      "currency": "USD"
    },
    {
      "tld": "dev",
```

**Errores comunes:**

| Código | Causa | Solución |

| 502 | Proveedor de dominios temporalmente no disponible | Reintenta después de unos segundos |
| 503 | Servicio de dominios deshabilitado | Contacta a soporte |

### Paso 5 — Registrar un dominio

**Objetivo:** Registrar un nuevo dominio para el usuario autenticado.

**Endpoint:** `POST /api/v1/domains`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/domains" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "example-demo.com",
    "years": 1,
    "whois_privacy": true,
    "auto_renew": false,
    "nameservers": null
  }'
```

**Respuesta (201):**

```json
{
  "success": true,
  "data": {
    "id": "507f1f77bcf86cd799439011",
    "domain": "example-demo.com",
    "status": "active",
    "expires_at": "2027-03-18T10:30:00",
    "created_at": "2026-03-18T10:30:00",
    "auto_renew": false,
    "whois_privacy": true,
    "locked": true,
    "nameservers": [
      "ns1.default-dns.com",
      "ns2.default-dns.com"
    ]
  },
  "msg": "Domain registered successfully"
}
```

:::note
Guarda `data.id` como `$DOMAIN_ID` para las solicitudes siguientes.
:::

:::warning
El registro de dominio descuenta del saldo del usuario de inmediato. Verifica la disponibilidad (Paso 3) y el saldo suficiente antes de llamar a este endpoint.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 400 | Nombre de dominio inválido | Verifica el formato del dominio (p. ej., `example.com`, máx. 253 caracteres) |
| 401 | Se requiere autenticación | Regenera los tokens |
| 402 | Saldo insuficiente | Agrega fondos a la cuenta del usuario antes de reintentar |
| 404 | Contacto no configurado | Crea primero un contacto de dominio (Paso 2) |
| 409 | Dominio no disponible o ya registrado | Verifica primero la disponibilidad (Paso 3), o el dominio ya está en la cuenta del usuario |
| 422 | Error de validación en el cuerpo de la solicitud | Verifica los formatos y restricciones de los campos |
| 429 | Límite de tasa excedido o límite de dominios alcanzado | Espera y reintenta, o verifica los límites de dominios de la cuenta |
| 502 | El proveedor de dominios no pudo procesar | Reintenta después de unos segundos |
| 503 | Servicio de dominios deshabilitado | Contacta a soporte |

### Paso 9 — Establecer registros DNS

**Objetivo:** Reemplazar todos los registros DNS de un dominio con un nuevo conjunto de registros.

**Endpoint:** `PUT /api/v1/domains/{domain_id}/dns`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X PUT "$BASE_URL/domains/$DOMAIN_ID/dns" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "records": [
      {
        "type": "A",
        "host": "@",
        "value": "93.184.216.34",
        "ttl": 3600
      },
      {
        "type": "CNAME",
        "host": "www",
        "value": "example-demo.com",
        "ttl": 3600
      },
      {
        "type": "MX",
        "host": "@",
        "value": "mail.example-demo.com",
        "ttl": 3600,
        "mx_pref": 10
      },
      {
        "type": "TXT",
        "host": "@",
        "value": "v=spf1 include:_spf.google.com ~all",
        "ttl": 3600
      }
    ]
  }'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "records": [
      {
        "type": "A",
        "host": "@",
        "value": "93.184.216.34",
        "ttl": 3600,
        "mx_pref": null
      },
      {
        "type": "CNAME",
        "host": "www",
        "value": "example-demo.com",
        "ttl": 3600,
        "mx_pref": null
      },
      {
        "type": "MX",
        "host": "@",
        "value": "mail.example-demo.com",
        "ttl": 3600,
        "mx_pref": 10
      },
      {
        "type": "TXT",
        "host": "@",
        "value": "v=spf1 include:_spf.google.com ~all",
        "ttl": 3600,
        "mx_pref": null
      }
    ]
  },
  "msg": "DNS records updated"
}
```

:::warning
Este endpoint **reemplaza todos los registros existentes**. Incluye siempre todos los registros deseados en la solicitud, no solo los que quieres agregar o cambiar. Obtén primero los registros actuales (Paso 8) para evitar eliminar accidentalmente registros existentes.
:::

:::tip
Los registros MX requieren el campo `mx_pref`. Para otros tipos de registro, `mx_pref` puede omitirse o establecerse en `null`.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 400 | Registro DNS inválido (tipo, host, valor o TTL incorrectos) | Verifica el formato del registro -- comprueba el enum `type`, la longitud de `host`, la longitud de `value` y el rango de `ttl` (60-86400) |
| 401 | Se requiere autenticación | Regenera los tokens |
| 404 | Dominio no encontrado | Verifica el `domain_id` |
| 422 | Error de validación | Verifica que el array `records` tenga de 1 a 100 elementos y todos los campos requeridos |
| 429 | Límite de tasa excedido | Espera y reintenta |
| 502 | Error del proveedor DNS | Reintenta después de unos segundos |

### Paso 10 — Establecer nameservers del dominio

**Objetivo:** Cambiar entre nameservers por defecto y personalizados para un dominio.

**Endpoint:** `PUT /api/v1/domains/{domain_id}/nameservers`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X PUT "$BASE_URL/domains/$DOMAIN_ID/nameservers" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "type": "custom",
    "nameservers": [
      "ns1.cloudflare.com",
      "ns2.cloudflare.com"
    ]
  }'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "type": "custom",
    "nameservers": [
      "ns1.cloudflare.com",
      "ns2.cloudflare.com"
    ]
  },
  "msg": "Nameservers updated"
```

## Ejemplo completo de extremo a extremo

Flujo completo con valores ficticios consistentes (Pasos 0-9: auth, contacto, verificación, precios, registro, listado, detalles, obtener DNS, establecer DNS).

```bash
BASE_URL="https://api.1platform.pro/api/v1"

# === PASO 0: Obtener el token de aplicación ===

  -H "Content-Type: application/json" \

# === PASO 1: Obtener el token de usuario ===
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "sk-user-carlos-2026"}' \
  | jq -r '.data.access_token')

echo "USER_TOKEN: $USER_TOKEN"

# === PASO 2: Crear el contacto de dominio ===
curl -s -X PUT "$BASE_URL/domains/contact" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "first_name": "Carlos",
    "last_name": "Ejemplo Demo",
    "email": "carlos@example.com",
    "phone": "+502.00001234",
    "address": "Avenida Reforma 10-50",
    "city": "Guatemala",
    "state": "Guatemala",
    "country": "GT",
    "postal_code": "01010"
  }' | jq .

# === PASO 3: Verificar disponibilidad de dominio ===
curl -s -X GET "$BASE_URL/domains/check?domain=example-demo.com" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .

curl -s -X GET "$BASE_URL/domains/pricing?tld=com&tld=io" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .

  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
```

## Idempotencia y buenas prácticas

### Idempotencia

:::warning[Sin mecanismo de idempotencia]
La API no admite un header `Idempotency-Key`. Si una solicitud de registro de dominio falla por timeout, lista los dominios del usuario (`GET /domains`) antes de reintentar para evitar cargos duplicados. El mismo dominio no puede registrarse dos veces -- la API devuelve un error **409**.
:::

### Reintentos
- Si recibes **429** (límite de tasa), espera el tiempo indicado y reintenta.
- Si recibes **500** o un **timeout**, verifica el estado del recurso antes de reintentar.
- Para **401** (token expirado), regenera el token y reintenta la solicitud original.
- Para **502** (proveedor no disponible), espera unos segundos y reintenta -- el proveedor externo puede estar temporalmente sobrecargado.

### Timeouts
- Las verificaciones de disponibilidad de dominio suelen completarse en **2-5 segundos**.
- El registro de dominio puede tardar **5-15 segundos** según el TLD y el proveedor.
- Las actualizaciones de registros DNS suelen completarse en **2-5 segundos**.
- Las transferencias de dominio son asíncronas y tardan **5-7 días** -- consulta periódicamente el endpoint de estado de la transferencia.
- Configura tu cliente HTTP con al menos **30 segundos** de timeout para los endpoints de registro y renovación.

### Tokens
- Ambos tokens (de aplicación y de usuario) expiran en **30 minutos** (1800 segundos). Implementa la renovación automática antes de la expiración.
- Ambos tokens son requeridos en todos los endpoints de gestión de dominios.

### Validaciones importantes
- Un contacto de dominio **debe** configurarse antes de registrar dominios o iniciar transferencias.
- Los nombres de dominio deben ser FQDNs válidos (máx. 253 caracteres).
- Los números de teléfono deben usar el formato E.164 con punto: `+N.NNNNNNN` (p. ej., `+502.00001234`).
- Los códigos de país deben tener exactamente 2 caracteres (ISO 3166-1 alpha-2).
- Los registros DNS se reemplazan por completo en cada `PUT` -- incluye siempre todos los registros deseados.
- Los registros MX requieren el campo `mx_pref`.
- Los cambios de nameserver surten efecto en minutos, pero pueden tardar hasta 48 horas en propagarse globalmente.
- Los cargos por registro y renovación de dominios no son reembolsables.
