# Flow: Generate Invoice

## Overview

This flow documents the step-by-step process to issue an electronic invoice (FEL) through the 1Platform API. The invoice is registered in the Guatemalan tax system via TribuTax integration.

**Use case:** Issue electronic invoices from your application, associated with a previously configured Business.

**End result:** An invoice with `issued` status, registered in TribuTax with its FEL UUID, series, fiscal number, and full FEL report metadata (receiver, issuer, certification, taxes, etc.).

---

## Prerequisites

| Requirement | Details |
|---|---|
| App API Key | Application key (`ak-...`) provided when registering your app |
| User API Key | User key (`sk-...`) obtained when creating the user |
| TribuTax configured | The App must have `keys.tributax` configured with `api_url`, `token`, and `login` (stored in the App document, not env vars) |
| Business created | You need a business with a valid `documentNumber` (NIT) |

### Required headers for all requests

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

## Data required to create an invoice

### Receptor (customer)

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | Yes | Legal name of the receptor |
| `documentType` | string | Yes | Document type (e.g., `"NIT"`, `"CUI"`, `"Pasaporte"`) |
| `documentNumber` | string | Yes | Document number (NIT without hyphen) |
| `email` | string | Yes | Receptor email (for invoice delivery) |
| `address` | string | No | Receptor address (default: `"Ciudad"`) |

### Items (invoice line items)

| Field | Type | Required | Description |
|---|---|---|---|
| `description` | string | Yes | Description of the good or service |
| `unit_price` | float | Yes | Unit price |
| `quantity` | int | Yes | Quantity |
| `discounts` | float | No | Applied discounts (default: `0`) |
| `type` | string | No | `"S"` = Service, `"B"` = Good (default: `"S"`) |

### Invoice options

| Field | Type | Required | Description |
|---|---|---|---|
| `items` | array | Yes | List of items (minimum 1) |
| `receptorInfo` | object | Yes | Receptor data |
| `date` | string | No | Date in `YYYY-MM-DD` format (default: today) |
| `emitOnCreate` | bool | No | Emit via TribuTax on creation (default: `true`) |
| `currency` | string | No | Currency (default: `"GTQ"`) |
| `description` | string | No | General invoice description |

:::warning
`branchId` is no longer accepted. If sent, the API returns a **422** error. Invoices are associated directly to the business via the `businessId` in the URL path.
:::

## Step-by-step flow

### Step 0 — Get App Token

**Goal:** Authenticate the application and obtain the app JWT token.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** None (this is the entry point)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Response (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Save `data.access_token` as `$APP_TOKEN`. Expires in 30 minutes.
:::

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 401 | Invalid API key or inactive app | Verify that `apiKey` is correct and the app is active |
| 422 | Missing or malformed `apiKey` field | Send `{"apiKey": "..."}` in the body |
| 429 | Rate limit exceeded (10 req/min) | Wait and retry |

### Step 1 — Get User Token

**Goal:** Authenticate the user and obtain the user JWT token.

**Endpoint:** `POST /api/v1/users/token`

**Auth:** `Authorization: Bearer $APP_TOKEN` (app token only, no user token required)

```bash
curl -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$USER_API_KEY'"}'
```

**Response (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "User token generated successfully"
}
```

:::note
Save `data.access_token` as `$USER_TOKEN`. Expires in 30 minutes.
:::

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 401 | Invalid user API key or inactive user | Verify the user's `apiKey` |
| 422 | Missing `apiKey` field | Send `{"apiKey": "..."}` in the body |
| 429 | Rate limit exceeded | Wait and retry |

:::tip
If you don't have a user yet, create one with `POST /api/v1/users` sending `{"email": "user@example.com"}`. The response will include the user's `api_key`.
:::

### Step 2 — Create Business

**Goal:** Register the invoice-issuing business. You only need to do this once.

**Endpoint:** `POST /api/v1/businesses`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/businesses" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "My Company S.A.",
    "documentType": "NIT",
    "documentNumber": "12345678",
    "address": "Zona 10, Guatemala City"
  }'
```

**Response (201):**

```json
{
  "success": true,
  "data": {
    "id": "507f1f77bcf86cd799439011",
    "name": "My Company S.A.",
    "documentType": "NIT",
    "documentNumber": "12345678",
    "address": "Zona 10, Guatemala City",
    "status": "active",
    "created_at": "2026-02-17T10:00:00Z",
    "updated_at": "2026-02-17T10:00:00Z"
  },
  "msg": "Business created successfully"
}
```

:::note
Save `data.id` as `$BUSINESS_ID`.
:::

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 401 | Invalid or expired token | Regenerate tokens (Steps 0 and 1) |
| 409 | A Business with that `documentNumber` already exists | Use the existing Business; retrieve it with `GET /api/v1/businesses/{businessId}` |
| 422 | Missing required fields | Verify that `name`, `documentType`, `documentNumber`, and `address` are present |

### Step 6 — Cancel Invoice

**Goal:** Cancel an issued invoice. If it was registered in TribuTax, it will also be voided there. After cancellation, the FEL report is re-fetched to capture updated cancellation data from FEL (`fel_cancellation`).

**Endpoint:** `DELETE /api/v1/businesses/{businessId}/invoices/{invoiceId}`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X DELETE "$BASE_URL/businesses/$BUSINESS_ID/invoices/$INVOICE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "reason": "Error in receptor data"
  }'
```

**Response (200):**

```json
{
  "success": true,
  "data": {
    "id": "709f1f77bcf86cd799439033",
    "status": "cancelled",
    "cancellation_reason": "Error in receptor data",
    "cancelled_at": "2026-02-17T12:00:00Z",
    "updated_at": "2026-02-17T12:00:00Z"
  },
  "msg": "Invoice cancelled successfully"
}
```

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 400 | Invoice is already cancelled | Cannot cancel twice |
| 401 | Invalid or expired token | Regenerate tokens |
| 404 | Business or Invoice not found | Verify the IDs |
| 422 | TribuTax error during cancellation | Verify the invoice has a valid `externalId`. Contact support if it persists |

## Full end-to-end example

Complete flow with consistent fictitious values, from authentication to invoice issuance.

```bash
# === VARIABLES ===
BASE_URL="https://api.1platform.pro/api/v1"

# === STEP 0: Get App Token ===
APP_TOKEN=$(curl -s -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "ak-prod-app-key-2026"}' \
  | jq -r '.data.access_token')

echo "APP_TOKEN: $APP_TOKEN"

# === STEP 1: Get User Token ===
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "sk-user-maria-2026"}' \
  | jq -r '.data.access_token')

echo "USER_TOKEN: $USER_TOKEN"

# === STEP 2: Create Business ===
BUSINESS_ID=$(curl -s -X POST "$BASE_URL/businesses" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Global Technology S.A.",
    "documentType": "NIT",
    "documentNumber": "55667788",
    "address": "Boulevard Los Proceres 15-80, Zone 10, Guatemala"
  }' \
  | jq -r '.data.id')

echo "BUSINESS_ID: $BUSINESS_ID"

# === STEP 3: Create Invoice ===
INVOICE_RESPONSE=$(curl -s -X POST "$BASE_URL/businesses/$BUSINESS_ID/invoices" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "items": [
      {
        "description": "Mobile app development - Phase 1",
        "unit_price": 15000.00,
        "quantity": 1,
        "type": "S"
      },
      {
        "description": "Annual hosting and domain",
        "unit_price": 1500.00,
        "quantity": 1,
        "discounts": 150.00,
        "type": "S"
      }
    ],
    "receptorInfo": {
      "name": "El Exito Distribution S.A.",
      "documentType": "NIT",
      "documentNumber": "11223344",
      "email": "invoices@elexito.com.gt",
      "address": "6th Street 3-25, Zone 4, Quetzaltenango"
    },
    "date": "2026-02-17",
    "emitOnCreate": true,
    "currency": "GTQ",
    "description": "Development and hosting services - February 2026"
  }')

echo "INVOICE_RESPONSE: $INVOICE_RESPONSE"

INVOICE_ID=$(echo $INVOICE_RESPONSE | jq -r '.data.id')
echo "INVOICE_ID: $INVOICE_ID"

# === STEP 4: Verify Invoice ===
curl -s -X GET "$BASE_URL/businesses/$BUSINESS_ID/invoices/$INVOICE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .
```

| Send invoice by email | Not implemented | The receptor's `email` field is sent to TribuTax, but there is no explicit endpoint to resend |
| Re-emit failed invoice | Not implemented | If an invoice was saved with `status: "failed"`, there is no endpoint to retry emission. A new invoice must be created |
| Assign/unassign users to Branch | Not implemented | Only the creator is automatically assigned. No endpoint exists to manage additional assignments |
| Update Business or Branch | Not implemented | No PUT/PATCH endpoints exist to modify businesses or branches |
| List user's Businesses | Not implemented | No endpoint to list all businesses. The user must know the `businessId` |

## Idempotency and best practices

### Idempotency

:::warning[No idempotency mechanism]
The API does not support an `Idempotency-Key` header. If a request fails due to timeout, check with `GET /invoices` before retrying to avoid duplicate invoices.
:::

### Retries

- If you receive **429** (rate limit), wait the indicated time and retry.
- If you receive **500** or a **timeout**, **do not retry creation immediately**. First query `GET /invoices?status=issued` to verify whether the invoice was created. Invoices that failed in TribuTax are saved with `status: "failed"`.
- For **401** (expired token), regenerate the token and retry the original request.

### Timeouts

:::warning
Invoice emission involves an external call to TribuTax with a configurable timeout (default: **30 seconds**, set in `App.keys.tributax.timeout`). Additionally, the FEL report is fetched from FEEL after emission. Configure your HTTP client with at least **60 seconds** timeout for the create endpoint.
:::

### Tokens

- Both tokens (app and user) expire in **30 minutes** (1800 seconds). Implement automatic renewal before expiration.
- Both tokens are required on all invoicing endpoints.

### Important validations

- The Business `documentNumber` must match the NIT registered in TribuTax.
- The receptor's `documentNumber` must be a valid NIT for Guatemalan electronic invoicing.
- At least **1 item** is required in the invoice.
- Do **not** send `branchId` in the request body — it will be rejected with a 422 error.
