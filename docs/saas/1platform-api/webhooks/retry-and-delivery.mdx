# Retry & Delivery

## Delivery guarantees

1Platform delivers outbound webhooks with **at-least-once** semantics. Your
handler **must** be idempotent — see
[Receiving notifications → idempotency](./receiving-notifications#idempotency-on-the-merchant-side).

## Retry schedule

When 1Platform receives any non-success response (or the request fails entirely),
it retries with exponential backoff:

| Attempt | Delay before retry | Cumulative time |
|---|---|---|
| 1 | — | 0 |
| 2 | 5 s | 5 s |
| 3 | 30 s | 35 s |
| 4 | 120 s | 155 s |

After the fourth attempt fails, the webhook is logged as
`payment_webhook.exhausted_retries` and dropped. There is no automatic
dead-letter queue today — operators must reconcile via
`GET /users/transactions` and a manual replay.

:::caution Plan for outages
If your backend is down for more than ~3 minutes, expect to lose some webhook
deliveries. Periodically reconcile with
`GET /users/transactions?from=...&to=...` to backfill missed events.
:::

## Response codes 1Platform expects

| Status | Behaviour |
|---|---|
| `2xx` | Success. No retry. |
| `3xx` | **Treated as failure** — your handler must not redirect, even on duplicates. |
| `4xx` | **Treated as failure and retried.** Even though the request is "your fault", 1Platform retries because a misconfigured `Authorization` header should not silently drop the event. |
| `5xx` | Retried (this is the expected path for transient failures). |
| Connection refused / DNS failure / timeout | Retried. |

:::tip Always return 200 quickly
Acknowledge receipt with `200 OK` (or `202 Accepted`) and a tiny body **before**
running heavy work. Defer fulfilment to a background queue so a slow handler
doesn't trigger spurious retries that amplify load during outages.
:::

## Timeouts

The dispatcher uses a **10-second** request timeout per attempt. If your
handler takes longer than that, the request will fail and 1Platform will retry
according to the schedule above.

## Concurrency

The dispatcher caps in-flight outbound webhooks at **20 concurrent dispatches
per process**. Webhooks beyond this cap wait up to 5 s for a slot before being
dropped — drop events show up as `payment_webhook.semaphore_timeout` in the
logs and are **not** retried.

This cap protects 1Platform from memory exhaustion when a downstream merchant
is slow to respond. If you need higher throughput, contact your account manager.

## Headers on every retry

| Header | Notes |
|---|---|
| `X-Webhook-Signature` | Recomputed every attempt (timestamp changes between retries) |
| `X-Webhook-Timestamp` | Also recomputed — verify the replay window against this value |
| `Content-Type: application/json` | Always |
| Custom headers from `body_template`'s `headers` field | Forwarded if listed in the allowlist (`authorization`, `content-type`, `x-idempotency-key`, `x-request-id`, `x-webhook-signature`, `x-webhook-timestamp`) |

## Internal vs external URLs

If the configured URL points to a 1platform.pro domain (e.g. the internal
billing-credit endpoint), the dispatcher **does not retry** and uses a
short-lived bearer token instead of HMAC. External URLs always use HMAC + the
retry schedule above.
