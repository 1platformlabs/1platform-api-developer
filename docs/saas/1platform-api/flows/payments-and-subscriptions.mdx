# Flow: Payments & Subscriptions

## Overview

This flow documents how to manage payments and subscriptions through the 1Platform API. It covers checking billing balance, creating payment transactions, viewing transaction history, retrieving subscription details, and tracking API consumption.

**Use case:** Allow users to top up their account balance via the integrated payment service, track transaction history, and manage their subscription plan.

**End result:** A completed payment transaction with balance credited to the user's account, full visibility into billing, subscription status, and detailed consumption tracking.

---

## Prerequisites

| Requirement | Details |
|---|---|
| App API Key | Application key (`ak-...`) — see [Environments](../reference/environments) |
| User API Key | User key (`sk-...`) obtained when creating the user |
| Payment gateway configured | **The user's billing must have an active gateway** — see callout below |

:::warning Payment gateway must be configured
`POST /users/transactions` returns **`422`** with `data.code: "PAYMENT_GATEWAY_NOT_CONFIGURED"`
when the user does not have an active gateway.

To configure one:

1. Provision a gateway via the operator dashboard (the operator action lives
   outside the API — contact your account manager for the QA dashboard URL).
2. Verify with `GET /users/billing` — `data.billing.payment_gateway_id` must
   be **non-null**.

The **merchant currency** is read from the App's configuration, not from the
request body. To charge in a different currency, register a separate App.
:::

### Required headers for all requests

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

## Data required to create a transaction

### Transaction

| Field | Type | Required | Description |
|---|---|---|---|
| `amount` | number | Yes | Transaction amount (must be positive) |
| `description` | string | No | Transaction description (max 500 chars) |
| `metadata` | object | No | Additional metadata (key-value pairs). **Echoed back unchanged** in the outbound webhook payload. |
| `merchant_reference` | string | No | Free-form merchant identifier (max 128 chars). **Echoed back unchanged** in the outbound webhook — set this to your local order ID for deterministic correlation. |

## Step-by-step flow

### Step 0 — Get App Token

**Goal:** Authenticate the application and obtain the app JWT token.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** None (this is the entry point)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Response (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Save `data.access_token` as `$APP_TOKEN`. Expires in 30 minutes.
:::

### Token caching pattern

Both tokens have a **30-minute TTL** (`expires_in: 1800`). Recommended client behaviour:

1. **Cache** the token after minting, with expiry = `expires_in - 60` (60-second safety margin).
2. **Pre-emptive refresh:** when within the safety margin, mint a fresh token before the next request.
3. **Reactive 401:** on a 401, mint a fresh token and retry the original request **once** (not twice — that masks real auth bugs).
4. **Per-user vs central:** the App token can be cached centrally (one shared cache). The User token is scoped to one user — cache per-user.

The auth endpoints are rate-limited to **10 / min**, so do not mint on every call.

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 401 | Invalid API key or inactive app | Verify that `apiKey` is correct and the app is active |
| 422 | Missing or malformed `apiKey` field | Send `{"apiKey": "..."}` in the body |
| 429 | Rate limit exceeded (10 req/min) | Wait and retry |

### Step 3 — Create Payment Transaction

**Goal:** Create a payment transaction and obtain a checkout URL where the user completes payment.

**Endpoint:** `POST /api/v1/users/transactions`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/users/transactions" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "amount": 10.00,
    "description": "Account top-up - March 2026",
    "metadata": {
      "source": "web",
      "campaign_id": "spring-promo-2026"
    }
  }'
```

**Response (201):**

```json
{
  "success": true,
  "data": {
    "id": "65a1b2c3d4e5f6789abcdef0",
    "user_id": "507f1f77bcf86cd799439011",
    "usd_amount": 10.00,
    "amount": 78.00,
    "currency": "USD",
    "currency_provider": "GTQ",
    "status": "created",
    "payment_url": "https://checkout.payment-service.com/pay/abc123",
    "reference": null,
    "description": "Account top-up - March 2026",
    "created_at": "2026-03-18T10:30:00Z",
    "updated_at": "2026-03-18T10:30:00Z"
  },
  "msg": "Transaction created — redirect user to payment URL"
}
```

:::note
Redirect the user to `data.payment_url` to complete the payment. Save `data.id` as `$TRANSACTION_ID`.
:::

**Key fields:**

- `usd_amount` — amount you sent (in your **merchant currency**, e.g. USD).
  **Use this for ledgering / crediting sellers.**
- `amount` — amount the upstream provider will charge (in **provider
  currency**, e.g. GTQ — converted by 1Platform at request time). Informational
  only; the exchange rate is opaque.
- `currency_provider` — the provider currency for `amount` (e.g. `GTQ`).
- `payment_url` — checkout URL where the user completes payment. Valid for a
  limited time — redirect promptly.
- `status` — starts as `"created"`. Final state arrives via webhook —
  see [receiving notifications](../webhooks/receiving-notifications).

:::warning No idempotency key on this endpoint
`POST /users/transactions` does **not** accept an `Idempotency-Key` header.
If your call times out, you may end up with a duplicate transaction at the
upstream provider.

Workaround:
1. Set `merchant_reference` on every create call.
2. Before retrying after a timeout, query
   `GET /users/transactions?metadata.merchant_reference=<your_ref>&limit=1`
   to reconcile.
3. If a transaction with your `merchant_reference` exists with status
   `created` or `approved`, treat the original call as successful.
:::

**Common errors:**

| Code | `data.code` | Cause | Solution |
|---|---|---|---|
| 400 | — | Invalid request data | Verify `amount` is a positive number |
| 401 | — | Authentication required | Regenerate tokens. See [token caching](#token-caching-pattern) |
| 404 | — | Billing record not found for user | Ensure the user has a billing record (created during user registration) |
| 422 | `PAYMENT_GATEWAY_NOT_CONFIGURED` | Tenant has no active gateway | Configure a gateway first — see prerequisite warning above |
| 422 | (Pydantic `detail`) | Schema validation error | Fix request body |
| 429 | — | Rate limit exceeded (20 req/min) | Wait and retry |
| 502 | — | Payment provider temporarily unavailable | Retry after a short delay (max 3 attempts, exponential backoff) |

### Refunds

`POST /api/v1/users/transactions/{transaction_id}/refund` is **reserved but
not yet implemented**. It always returns:

```json
{
  "success": false,
  "data": { "code": "REFUND_NOT_SUPPORTED" },
  "msg": "Refund via API is not yet supported. Issue refunds via the operator dashboard until this endpoint is implemented."
}
```

HTTP status: **`501 Not Implemented`**. Clients should not retry. To refund a
transaction in the meantime, issue it via the operator dashboard. There is
**no** automatic outbound webhook for refunds today.

## Full end-to-end example

Complete flow with consistent fictitious values.

```bash
# === VARIABLES ===
BASE_URL="https://api.1platform.pro/api/v1"

# === STEP 0: Get App Token ===
APP_TOKEN=$(curl -s -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "ak-prod-app-key-2026"}' \
  | jq -r '.data.access_token')

echo "APP_TOKEN: $APP_TOKEN"

# === STEP 1: Get User Token ===
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "sk-user-maria-2026"}' \
  | jq -r '.data.access_token')

echo "USER_TOKEN: $USER_TOKEN"

# === STEP 2: Check Billing Info ===
BILLING_RESPONSE=$(curl -s -X GET "$BASE_URL/users/billing" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN")

echo "Current balance: $(echo $BILLING_RESPONSE | jq '.data.billing.balance')"

SUBSCRIPTION_ID=$(echo $BILLING_RESPONSE | jq -r '.data.subscription.id')
echo "SUBSCRIPTION_ID: $SUBSCRIPTION_ID"

# === STEP 3: Create Payment Transaction ===
TRANSACTION_RESPONSE=$(curl -s -X POST "$BASE_URL/users/transactions" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "amount": 25.00,
    "description": "Account top-up"
  }')

PAYMENT_URL=$(echo $TRANSACTION_RESPONSE | jq -r '.data.payment_url')
echo "Redirect user to: $PAYMENT_URL"

# === STEP 4: Get Transaction History ===
curl -s -X GET "$BASE_URL/users/transactions?limit=5" \
  -H "Authorization: Bearer $APP_TOKEN" \

  | jq .

# === STEP 6: List Available Plans ===
curl -s -X GET "$BASE_URL/plans" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .

# === STEP 7: Get Consumption History ===
curl -s -X GET "$BASE_URL/users/consumption?limit=5" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .

# === STEP 8: Get Consumption Summary ===
curl -s -X GET "$BASE_URL/users/consumption/summary" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .
```

### Quick rules of thumb

- **Timeouts.** Transaction creation involves an external call and may take
  **5–15 seconds**. Set a **30s** client timeout.
- **Retries on 5xx.** Exponential backoff, max 3 attempts. **Never** retry
  blindly without setting `merchant_reference`.
- **`amount` must be positive.** `description` has a 500-char max.
- **`payment_url` expires fast** — redirect immediately, do not stash it.
- **Webhooks fire on terminal states only** (`approved` / `denied` /
  `cancelled` / `expired` / `dismiss`). The `created` state never fires a
  webhook.

## Quick endpoint reference

| Step | Method | Endpoint | Description |
|---|---|---|---|
| 0 | `POST` | `/api/v1/auth/token` | Get app token |
| 1 | `POST` | `/api/v1/users/token` | Get user token |
