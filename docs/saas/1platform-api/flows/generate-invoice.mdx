# Flujo: Generar factura

## Vista general

Este flujo documenta el proceso paso a paso para emitir una factura electrónica (FEL) a través de la 1Platform API. La factura se registra en el sistema tributario guatemalteco mediante la integración con TribuTax.

**Caso de uso:** Emitir facturas electrónicas desde tu aplicación, asociadas a un Business previamente configurado.

**Resultado final:** Una factura con estado `issued`, registrada en TribuTax con su FEL UUID, serie, número fiscal y los metadatos completos del reporte FEL (receptor, emisor, certificación, impuestos, etc.).

---

## Requisitos previos

| Requisito | Detalles |
|---|---|
| App API Key | Clave de aplicación (`ak-...`) proporcionada al registrar tu app |
| User API Key | Clave de usuario (`sk-...`) obtenida al crear el usuario |
| TribuTax configurado | La App debe tener `keys.tributax` configurado con `api_url`, `token` y `login` (almacenado en el documento de la App, no en variables de entorno) |
| Business creado | Necesitas un business con un `documentNumber` (NIT) válido |

### Headers requeridos para todas las solicitudes

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

---

## Variables usadas en los ejemplos

```bash
BASE_URL="https://api.1platform.pro/api/v1"
APP_API_KEY="ak-your-app-api-key"
USER_API_KEY="sk-user-abc123"
APP_TOKEN=""    # Obtenido en el Paso 0
USER_TOKEN=""   # Obtenido en el Paso 1
BUSINESS_ID=""  # Obtenido en el Paso 2
BRANCH_ID=""    # Obtenido en el Paso 2b (opcional)
```

## Datos requeridos para crear una factura

### Receptor (cliente)

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `name` | string | Sí | Nombre legal del receptor |
| `documentType` | string | Sí | Tipo de documento (p. ej., `"NIT"`, `"CUI"`, `"Pasaporte"`) |
| `documentNumber` | string | Sí | Número de documento (NIT sin guion) |
| `email` | string | Sí | Correo del receptor (para el envío de la factura) |
| `address` | string | No | Dirección del receptor (por defecto: `"Ciudad"`) |

### Items (líneas de la factura)

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `description` | string | Sí | Descripción del bien o servicio |
| `unit_price` | float | Sí | Precio unitario |
| `quantity` | int | Sí | Cantidad |
| `discounts` | float | No | Descuentos aplicados (por defecto: `0`) |
| `type` | string | No | `"S"` = Servicio, `"B"` = Bien (por defecto: `"S"`) |

### Opciones de la factura

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `items` | array | Sí | Lista de items (mínimo 1) |
| `receptorInfo` | object | Sí | Datos del receptor |
| `date` | string | No | Fecha en formato `YYYY-MM-DD` (por defecto: hoy) |
| `emitOnCreate` | bool | No | Emitir vía TribuTax al crear (por defecto: `true`) |
| `currency` | string | No | Moneda (por defecto: `"GTQ"`) |
| `description` | string | No | Descripción general de la factura |

:::warning
`branchId` ya no se acepta. Si se envía, la API devuelve un error **422**. Las facturas se asocian directamente al business mediante el `businessId` en la ruta de la URL.
:::

## Flujo paso a paso

### Paso 0 — Obtener el token de aplicación

**Objetivo:** Autenticar la aplicación y obtener el token JWT de la app.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** Ninguna (este es el punto de entrada)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$APP_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key inválida o app inactiva | Verifica que `apiKey` sea correcto y que la app esté activa |
| 422 | Campo `apiKey` ausente o mal formado | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido (10 req/min) | Espera y reintenta |

---

### Paso 1 — Obtener el token de usuario

**Objetivo:** Autenticar al usuario y obtener el token JWT de usuario.

**Endpoint:** `POST /api/v1/users/token`

**Auth:** `Authorization: Bearer $APP_TOKEN` (solo token de aplicación, no se requiere token de usuario)

```bash
curl -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$USER_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "User token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$USER_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key de usuario inválida o usuario inactivo | Verifica el `apiKey` del usuario |
| 422 | Campo `apiKey` ausente | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido | Espera y reintenta |

:::tip
Si aún no tienes un usuario, crea uno con `POST /api/v1/users` enviando `{"email": "user@example.com"}`. La respuesta incluirá el `api_key` del usuario.
:::

---

### Paso 2 — Crear el Business

**Objetivo:** Registrar el business emisor de facturas. Solo necesitas hacerlo una vez.

**Endpoint:** `POST /api/v1/businesses`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/businesses" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "My Company S.A.",
    "documentType": "NIT",
    "documentNumber": "12345678",
    "address": "Zona 10, Guatemala City"
  }'
```

**Respuesta (201):**

```json
{
  "success": true,
  "data": {
    "id": "507f1f77bcf86cd799439011",
    "name": "My Company S.A.",
    "documentType": "NIT",
    "documentNumber": "12345678",
    "address": "Zona 10, Guatemala City",
    "status": "active",
    "created_at": "2026-02-17T10:00:00Z",
    "updated_at": "2026-02-17T10:00:00Z"
  },
  "msg": "Business created successfully"
}
```

:::note
Guarda `data.id` como `$BUSINESS_ID`.
:::

:::tip[Crea el business y su primera sucursal en una sola llamada]
Puedes incluir un objeto **`branch`** opcional en el cuerpo de la solicitud. Cuando está presente, la
API crea la sucursal **y** vincula al usuario que llama (como propietario de la sucursal) en la
misma llamada, de modo que el business es devuelto de inmediato por `GET /api/v1/businesses`. Omitir
`branch` crea un business independiente que **no** está vinculado a ningún usuario; agrega una sucursal
más adelante mediante [el Paso 2b](#step-2b--create-branch-optional) para que aparezca en los listados.

```bash
curl -X POST "$BASE_URL/businesses" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "My Company S.A.",
    "documentType": "NIT",
    "documentNumber": "12345678",
    "address": "Zona 10, Guatemala City",
    "branch": { "name": "Main Branch", "commerceId": "COM-001" }
  }'
```
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | Token inválido o expirado | Regenera los tokens (Pasos 0 y 1) |
| 409 | Ya existe un Business con ese `documentNumber` | Usa el Business existente; recupéralo con `GET /api/v1/businesses/{businessId}` |
| 422 | Faltan campos requeridos | Verifica que `name`, `documentType`, `documentNumber` y `address` estén presentes |

### Paso 2b — Crear sucursal (opcional) {#step-2b--create-branch-optional}
**Objetivo:** Registrar una sucursal bajo el business. Las sucursales tienen un `commerceId` requerido por TribuTax para business con múltiples establecimientos. Si tu business opera desde una sola ubicación, puedes omitir este paso.

**Endpoint:** `POST /api/v1/businesses/{businessId}/branches`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/businesses/$BUSINESS_ID/branches" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Sucursal Central",
    "commerceId": "COM-001"
  }'
```

**Respuesta (201):**

```json
{
  "success": true,
  "data": {
    "id": "607f1f77bcf86cd799439022",
    "business_id": "507f1f77bcf86cd799439011",
    "name": "Sucursal Central",
    "commerceId": "COM-001",
    "status": "active",
    "assigned_users": [],
    "created_at": "2026-02-17T10:05:00Z",
    "updated_at": "2026-02-17T10:05:00Z"
  },
  "msg": "Branch created successfully"
}
```

:::note
Guarda `data.id` como `$BRANCH_ID`. El `commerceId` es el identificador de comercio de TribuTax para esta sucursal.
:::

**Datos de la sucursal:**

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `name` | string | Sí | Nombre visible de la sucursal (máx. 255 caracteres) |
| `commerceId` | string | Sí | ID de comercio para la integración con TribuTax |

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | Token inválido o expirado | Regenera los tokens (Pasos 0 y 1) |
| 404 | Business no encontrado | Verifica el `businessId` en la URL |
| 422 | Faltan campos requeridos | Verifica que `name` y `commerceId` estén presentes |
| 429 | Límite de tasa excedido | Espera y reintenta |
| 500 | Error interno del servidor | Contacta a soporte |

### Paso 2d — Buscar persona por documento (opcional)

**Objetivo:** Recuperar los datos personales de una persona usando su número de CUI/DPI. Útil para pre-rellenar los campos del receptor de la factura (`name`, `documentNumber`) sin pedirle al cliente que los escriba manualmente.

**Endpoint:** `POST /api/v1/identify`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/identify" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "document_number": "2943817650101"
```

|---|---|---|---|

```json
    "marital_status": "SO",
    "nationality": "guatemalteca",
    "birth_country": "Guatemala",
    "birth_department": "Guatemala",

    "occupation": "Comerciante",
    "expiration_date": "15/08/2030"

```

:::note
Combina `first_name`, `second_name`, `first_surname` y `second_surname` para construir el campo `name` del receptor. El `document_number` de la respuesta es el CUI, que puede pasarse como `documentNumber` en el receptor de la factura con `documentType: "CUI"`.

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | Token inválido o expirado | Regenera los tokens (Pasos 0 y 1) |
| 404 | No se encontraron registros para el número de documento | Verifica que el CUI/DPI sea correcto (13 dígitos) |
| 422 | Integración no configurada o error en la búsqueda | Asegúrate de que la App tenga habilitada la integración de búsqueda de personas |
| 429 | Límite de tasa excedido (10 req/min) | Espera y reintenta |
| 500 | Error interno del servidor | Contacta a soporte |

### Paso 3 — Crear la factura

**Objetivo:** Emitir una factura electrónica (FEL) asociada a un business. El NIT del business se usa como emisor en TribuTax.

**Endpoint:** `POST /api/v1/businesses/{businessId}/invoices`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/businesses/$BUSINESS_ID/invoices" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{

      {
        "description": "Consulting service",
        "unit_price": 1000.00,
        "quantity": 1
```

| 500 | Error interno | Contacta a soporte. La factura puede haberse guardado con estado `failed` |

### Paso 4 — Obtener detalles de la factura

**Objetivo:** Recuperar los detalles completos de una factura emitida. La respuesta se lee directamente de la base de datos — los metadatos FEL ya se obtuvieron durante la creación (o la anulación).

**Endpoint:** `GET /api/v1/businesses/{businessId}/invoices/{invoiceId}`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X GET "$BASE_URL/businesses/$BUSINESS_ID/invoices/$INVOICE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

### Paso 6 — Anular factura

**Objetivo:** Anular una factura emitida. Si fue registrada en TribuTax, también se anulará allí. Tras la anulación, el reporte FEL se vuelve a obtener para capturar los datos de anulación actualizados de FEL (`fel_cancellation`).

**Endpoint:** `DELETE /api/v1/businesses/{businessId}/invoices/{invoiceId}`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X DELETE "$BASE_URL/businesses/$BUSINESS_ID/invoices/$INVOICE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "reason": "Error in receptor data"
  }'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "id": "709f1f77bcf86cd799439033",
    "status": "cancelled",
    "cancellation_reason": "Error in receptor data",
    "cancelled_at": "2026-02-17T12:00:00Z",
    "updated_at": "2026-02-17T12:00:00Z"
  },
  "msg": "Invoice cancelled successfully"
```

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 400 | La factura ya está anulada | No se puede anular dos veces |
| 401 | Token inválido o expirado | Regenera los tokens |
| 404 | Business o factura no encontrados | Verifica los IDs |
| 422 | Error de TribuTax durante la anulación | Verifica que la factura tenga un `externalId` válido. Contacta a soporte si persiste |

## Idempotencia y buenas prácticas

### Idempotencia

:::warning[Sin mecanismo de idempotencia]
La API no admite un header `Idempotency-Key`. Si una solicitud falla por timeout, verifica con `GET /invoices` antes de reintentar para evitar facturas duplicadas.
:::

### Reintentos

- Si recibes un **429** (límite de tasa), espera el tiempo indicado y reintenta.
- Si recibes un **500** o un **timeout**, **no reintentes la creación de inmediato**. Primero consulta `GET /invoices?status=issued` para verificar si la factura se creó. Las facturas que fallaron en TribuTax se guardan con `status: "failed"`.
- Para un **401** (token expirado), regenera el token y reintenta la solicitud original.

### Timeouts

:::warning
La emisión de facturas implica una llamada externa a TribuTax con un timeout configurable (por defecto: **30 segundos**, definido en `App.keys.tributax.timeout`). Además, el reporte FEL se obtiene de FEEL después de la emisión. Configura tu cliente HTTP con un timeout de al menos **60 segundos** para el endpoint de creación.
:::

### Tokens

- Ambos tokens (de aplicación y de usuario) expiran en **30 minutos** (1800 segundos). Implementa la renovación automática antes de la expiración.
- Ambos tokens son requeridos en todos los endpoints de facturación.

### Validaciones importantes

- El `documentNumber` del Business debe coincidir con el NIT registrado en TribuTax.
- El `documentNumber` del receptor debe ser un NIT válido para la facturación electrónica guatemalteca.
- Se requiere al menos **1 item** en la factura.
- **No** envíes `branchId` en el cuerpo de la solicitud — será rechazado con un error 422.
