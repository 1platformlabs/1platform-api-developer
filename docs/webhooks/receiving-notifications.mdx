# Receiving Notifications

When a transaction transitions to a terminal state, 1Platform fires a webhook
to the URL you configured for that event (see
[Configuring URLs](./configuring-urls)).

## Canonical payload

If you configured the event handler **without** a `body_template`, the payload
follows this canonical shape:

```json
{
  "event": "on_approved",
  "transaction_id": "65a1b2c3d4e5f6789abcdef0",
  "external_transaction_id": "MIGOGT123456",
  "status": "approved",
  "amount": 78.00,
  "currency": "GTQ",
  "merchant_amount": 10.00,
  "merchant_currency": "USD",
  "metadata": { "order_id": "...", "campaign_id": "..." },
  "merchant_reference": "bower_order_xyz",
  "reference": "GT123456",
  "timestamp": "2026-04-29T10:35:00+00:00"
}
```

If you configured a `body_template`, the dispatcher emits whatever you defined
— the canonical body is the **default** when you opt out of templating.

## Field reference

| Field | Origin | Use it for |
|---|---|---|
| `event` | 1Platform | Routing inside your handler when one URL serves all five events |
| `transaction_id` | 1Platform | Correlating with the `data.id` you stored at create time |
| `external_transaction_id` | upstream provider | Operator support tickets — the value the provider's dashboard shows |
| `status` | 1Platform | Final state of the transaction. See enum below |
| `amount` | provider currency | Informational only — the amount the upstream provider charged |
| `currency` | provider currency | The provider currency code (e.g. `GTQ`) |
| `merchant_amount` | **your** currency | **Use this for ledgering**. The amount you originally quoted |
| `merchant_currency` | your currency | E.g. `USD` |
| `metadata` | echoed | The free-form object you sent at create time |
| `merchant_reference` | echoed | The free-form ID you sent at create time. Use it as the deterministic key for idempotency |
| `reference` | upstream provider | Provider-internal reference (e.g. card auth ID) |
| `timestamp` | 1Platform | Last update time on the transaction (ISO 8601 with offset) |

:::warning amount vs merchant_amount
The `amount` field is in the **provider's** currency (often GTQ) because the
upstream payment processor converts USD to GTQ at charge time. **Never credit a
seller using `amount`** — use `merchant_amount` + `merchant_currency`. The
`amount` is informational; the exchange rate is opaque to the merchant.
:::

## Status enum

| Inbound `status` | Final? | Meaning |
|---|---|---|
| `created` | No | Transaction created, buyer has not paid yet (no webhook fires for this state) |
| `approved` | Yes | Buyer paid; funds settled |
| `denied` | Yes | Card declined |
| `cancelled` | Yes | Buyer cancelled before paying |
| `expired` | Yes | Payment URL expired |
| `dismiss` | Yes | Buyer closed the payment window |

## Merchant-side FSM mapping

Most marketplaces only need three states (`pending`, `paid`, `failed`). Map
the five terminal `status` values like this:

| Provider `status` | Merchant FSM state |
|---|---|
| `approved` | `paid` (and proceed to fulfilment) |
| `denied` | `failed` |
| `cancelled` | `failed` |
| `expired` | `failed` |
| `dismiss` | `failed` |

If you implement refunds out-of-band (via the operator dashboard), surface them
as an additional `refunded` state; there is no automatic webhook for refunds
yet. See [the dedicated note in payments-and-subscriptions](../flows/payments-and-subscriptions#refunds).

## Response wrapper differs from the rest of the API

Outbound webhook payloads are **not** wrapped in `{ success, data, msg }`. They
are flat objects (canonical or templated). This mirrors what most integrators
expect from a webhook receiver and stays compatible with HMAC signing over the
raw body. See [response format](../reference/response-format) for the full
rule.

## Idempotency on the merchant side

1Platform delivers webhooks at-least-once with retries — see
[Retry & delivery](./retry-and-delivery). Your handler **must** deduplicate.

The recommended deduplication key is `(transaction_id, event)`. If you set
`merchant_reference` at create time, you can also dedupe by
`(merchant_reference, event)` to align with your own order IDs.
