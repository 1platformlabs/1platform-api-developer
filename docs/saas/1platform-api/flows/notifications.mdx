# Flujo: Notificaciones

## Vista general

Este flujo documenta cómo un dashboard o una herramienta de administración se integra con el subsistema de notificaciones: listar el feed de notificaciones del usuario, difundir mensajes desde un contexto de administración, marcar entradas individuales como leídas y leer el conteo de no leídas para el icono de campana de la barra de navegación.

**Caso de uso:** Mostrar una bandeja de notificaciones in-app con badges de no leídas, y permitir que los administradores envíen anuncios a todos los usuarios o a un usuario específico.

**Resultado final:** El dashboard renderiza un feed de notificaciones actualizado con el estado leído/no leído, y el badge de no leídas se mantiene sincronizado con `/notifications/unread-count`.

---

## Requisitos previos

| Requisito | Detalles |
|---|---|
| App API Key | Clave de aplicación (`ak-...`) provista al registrar tu app |
| User API Key | Clave de usuario (`sk-...`) obtenida al crear el usuario |
| Rol de administrador | Crear notificaciones requiere privilegios de administrador en la app |

### Headers requeridos en todas las solicitudes

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

---

## Variables usadas en los ejemplos

```bash
BASE_URL="https://api.1platform.pro/api/v1"
APP_API_KEY="ak-your-app-api-key"
USER_API_KEY="sk-user-abc123"
APP_TOKEN=""           # Obtenido en el Paso 0
USER_TOKEN=""          # Obtenido en el Paso 1
NOTIFICATION_ID=""     # Obtenido en el Paso 2 o el Paso 3
```

## Modelo de datos

### Objeto Notification

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `id` | string | Sí | MongoDB ObjectId de la notificación |
| `title` | string | Sí | Titular corto que se muestra en el feed |
| `body` | string | Sí | Cuerpo completo de la notificación (se permite Markdown) |
| `severity` | string | Sí | Uno de `info`, `success`, `warning` |
| `link_url` | string \| null | Sí | URL opcional a la que enlaza la notificación |
| `read_at` | string \| null | Sí | Timestamp ISO-8601 de cuándo se marcó como leída, `null` si está sin leer |
| `created_at` | string | Sí | Timestamp ISO-8601 de cuándo se creó |

### Payload de creación

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `title` | string | Sí | Titular corto (se recomienda máx. ~120 caracteres) |
| `body` | string | No | Texto completo del cuerpo |
| `severity` | string | No | `info` (por defecto), `success` o `warning` |
| `link_url` | string \| null | No | URL opcional para click-through |
| `user_id` | string \| null | No | Apunta a un usuario específico. Mutuamente excluyente con `is_global` |
| `is_global` | boolean | No | Si es `true`, difunde a todos los usuarios de la app |

## Flujo paso a paso

### Paso 0 — Obtener el token de aplicación

**Objetivo:** Autenticar la aplicación y obtener el token JWT de la app.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** Ninguna (este es el punto de entrada)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$APP_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key inválida o app inactiva | Verifica que `apiKey` sea correcta y que la app esté activa |
| 422 | Campo `apiKey` ausente o malformado | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido (10 req/min) | Espera y reintenta |

---

### Paso 1 — Obtener el token de usuario

**Objetivo:** Autenticar al usuario y obtener el token JWT de usuario.

**Endpoint:** `POST /api/v1/users/token`

**Auth:** `Authorization: Bearer $APP_TOKEN` (solo el token de aplicación, no se requiere token de usuario)

```bash
curl -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$USER_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "User token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$USER_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key de usuario inválida o usuario inactivo | Verifica la `apiKey` del usuario |
| 422 | Campo `apiKey` ausente | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido | Espera y reintenta |

:::tip
Si aún no tienes un usuario, crea uno con `POST /api/v1/users` enviando `{"email": "user@example.com"}`. La respuesta incluirá la `api_key` del usuario.
:::

---

### Paso 2 — Listar notificaciones

**Objetivo:** Paginar el feed de notificaciones del usuario autenticado.

**Endpoint:** `GET /api/v1/notifications`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

**Parámetros de consulta:**

| Parámetro | Tipo | Requerido | Descripción |
|---|---|---|---|
| `unread_only` | boolean | No | Cuando es `true`, devuelve solo las notificaciones no leídas |
| `skip` | integer | No | Offset para la paginación (por defecto `0`) |
| `limit` | integer | No | Tamaño de página (por defecto `20`) |

```bash
curl -X GET "$BASE_URL/notifications?unread_only=false&skip=0&limit=20" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": [
    {
      "id": "507f1f77bcf86cd799439011",
      "title": "Generación de contenido completada",
      "body": "Tu artículo \"Guía de SEO 2026\" está listo.",
      "severity": "success",
      "link_url": "https://app.example.com/posts/507f1f77bcf86cd799439020",
      "read_at": null,
      "created_at": "2026-04-27T15:00:00Z"
    },
    {
      "id": "507f1f77bcf86cd799439012",
      "title": "Mantenimiento programado",
      "body": "El servicio estará en mantenimiento el 30/04 a las 02:00.",
      "severity": "warning",
      "link_url": null,
      "read_at": "2026-04-26T18:30:00Z",
      "created_at": "2026-04-26T18:00:00Z"
    }
  ],
  "msg": "Notifications fetched successfully"
}
```

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 422 | Tipo de parámetro de consulta inválido | Pasa `unread_only` como boolean y `skip`/`limit` como integers |
| 429 | Límite de tasa excedido | Espera y reintenta |

---

### Paso 3 — Crear una notificación (admin)

**Objetivo:** Enviar una notificación a un usuario específico o difundirla a todos los usuarios de la app.

**Endpoint:** `POST /api/v1/notifications`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/notifications" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Nueva integración disponible",
    "body": "Conecta tu cuenta de Google Search Console para ver insights.",
    "severity": "info",
    "link_url": "https://app.example.com/integrations/search-console",
    "is_global": true
  }'
```

**Respuesta (201):**

```json
{
  "success": true,
  "data": {
    "id": "507f1f77bcf86cd799439013",
    "title": "Nueva integración disponible",
    "body": "Conecta tu cuenta de Google Search Console para ver insights.",
    "severity": "info",
    "link_url": "https://app.example.com/integrations/search-console",
    "read_at": null,
    "created_at": "2026-04-27T15:30:00Z"
  },
  "msg": "Notification created successfully"
}
```

:::warning
`user_id` e `is_global: true` son mutuamente excluyentes: define uno u otro. Enviar ambos devuelve **422**.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 422 | Falta `title`, `severity` inválido, o conflicto entre `user_id` + `is_global` | Define exactamente un campo de destino y un `severity` válido |
| 429 | Límite de tasa excedido | Espera y reintenta |

---

### Paso 4 — Marcar una notificación como leída

**Objetivo:** Actualizar el timestamp `read_at` en una sola notificación.

**Endpoint:** `PATCH /api/v1/notifications/{notification_id}/read`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X PATCH "$BASE_URL/notifications/$NOTIFICATION_ID/read" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "id": "507f1f77bcf86cd799439011",
    "title": "Generación de contenido completada",
    "body": "Tu artículo \"Guía de SEO 2026\" está listo.",
    "severity": "success",
    "link_url": "https://app.example.com/posts/507f1f77bcf86cd799439020",
    "read_at": "2026-04-27T15:35:00Z",
    "created_at": "2026-04-27T15:00:00Z"
  },
  "msg": "Notification marked as read"
}
```

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 404 | La notificación no existe o pertenece a otro usuario | Verifica el `notification_id` |
| 422 | Parámetro de ruta `notification_id` malformado | Usa un ObjectId válido |
| 429 | Límite de tasa excedido | Espera y reintenta |

---

### Paso 5 — Leer el conteo de no leídas

**Objetivo:** Mostrar el badge de no leídas en el icono de campana: una verificación rápida y ligera.

**Endpoint:** `GET /api/v1/notifications/unread-count`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X GET "$BASE_URL/notifications/unread-count" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "unread": 2
  },
  "msg": "Unread count fetched successfully"
}
```

:::tip
Consulta este endpoint cada 30–60 segundos, o engánchalo a un webhook. Es mucho más económico que llamar a `/notifications?unread_only=true`, porque el servidor devuelve solo el conteo y no los documentos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 422 | Header de token de usuario ausente o malformado | Envía `x-user-token: $USER_TOKEN` |
| 429 | Límite de tasa excedido | Reduce la frecuencia de polling o aplica back off |

## Ejemplo completo de extremo a extremo

Flujo completo con valores ficticios consistentes.

```bash
# === VARIABLES ===
BASE_URL="https://api.1platform.pro/api/v1"

# === PASO 0: Obtener el token de aplicación ===
APP_TOKEN=$(curl -s -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "ak-prod-app-key-2026"}' \
  | jq -r '.data.access_token')

# === PASO 1: Obtener el token de usuario ===
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "sk-user-maria-2026"}' \
  | jq -r '.data.access_token')

# === PASO 2: Listar notificaciones ===
curl -s -X GET "$BASE_URL/notifications?unread_only=true&limit=10" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" | jq

# === PASO 3: Crear una notificación global ===
NOTIFICATION_ID=$(curl -s -X POST "$BASE_URL/notifications" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Demo notification",
    "body": "Sent from the Integration Flows guide",
    "severity": "info",
    "is_global": true
  }' | jq -r '.data.id')

echo "Notificación creada: $NOTIFICATION_ID"

# === PASO 4: Marcar como leída ===
curl -s -X PATCH "$BASE_URL/notifications/$NOTIFICATION_ID/read" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" | jq

# === PASO 5: Leer el conteo de no leídas ===
curl -s -X GET "$BASE_URL/notifications/unread-count" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" | jq
```

## Idempotencia y buenas prácticas

### Idempotencia
- `PATCH /notifications/{id}/read` es idempotente: marcar una notificación ya leída devuelve el mismo estado final.
- `POST /notifications` **no** es idempotente: cada llamada crea una nueva notificación. Envía un identificador de deduplicación en tu capa de aplicación si necesitas semántica de exactamente-una-vez.
- Los endpoints `GET` son de solo lectura y naturalmente idempotentes.

### Reintentos
- Si recibes **429** (límite de tasa), aplica back off y reintenta. El endpoint de unread-count es seguro de llamar cada 30–60 segundos.
- Para errores **5xx**, reintenta con backoff exponencial.
- Para **401**, regenera los tokens y reintenta.

### Timeouts
- Todos los endpoints son operaciones CRUD simples: configura un timeout de cliente de 5 segundos.

### Tokens
- Ambos tokens (de aplicación y de usuario) expiran en **30 minutos** (1800 segundos). Implementa una renovación automática antes de la expiración.

### Validaciones importantes
- `severity` debe ser uno de `info`, `success`, `warning`: la API rechaza cualquier otro valor.
- `user_id` e `is_global: true` son mutuamente excluyentes.
- `link_url` debería ser HTTPS: los navegadores pueden marcar las URLs http como inseguras.

## Referencia rápida de endpoints

| Paso | Método | Endpoint | Descripción |
|---|---|---|---|
| 0 | `POST` | `/api/v1/auth/token` | Obtener el token de aplicación |
| 1 | `POST` | `/api/v1/users/token` | Obtener el token de usuario |
| 2 | `GET` | `/api/v1/notifications` | Listar notificaciones (paginado) |
| 3 | `POST` | `/api/v1/notifications` | Crear una notificación (admin) |

| 5 | `GET` | `/api/v1/notifications/unread-count` | Leer el conteo de no leídas |
