# Configuring URLs

Webhook setup is a one-time bootstrap per environment. The full step-by-step
guide lives in [the Webhook Configuration flow](../flows/webhook-configuration).

## At-a-glance

| Step | Endpoint | Purpose |
|---|---|---|
| 1 | `GET /webhooks/config/` | Read current state |
| 2 | `POST /webhooks/config/domains` | Register a domain (returns the initial `webhook_secret` **once**) |
| 3 | `PUT /webhooks/config/events` | Map each `on_*` event to one of your URLs |
| 4 | `DELETE /webhooks/config/domains/{domain}` | Unregister a domain |
| 5 | `POST /webhooks/config/rotate-secret` | Rotate the signing secret (returns new value **once**) |

## Where each URL points

Bower's marketplace pattern: register one merchant domain (e.g.
`api.bowerbird.pro`), then map all five events to a single endpoint that uses
the canonical payload's `event` field for internal routing.

```bash
curl -X PUT "$BASE_URL/webhooks/config/events" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "on_approved":  { "url": "https://api.bowerbird.pro/webhooks/payments", "method": "POST" },
    "on_denied":    { "url": "https://api.bowerbird.pro/webhooks/payments", "method": "POST" },
    "on_cancelled": { "url": "https://api.bowerbird.pro/webhooks/payments", "method": "POST" },
    "on_expired":   { "url": "https://api.bowerbird.pro/webhooks/payments", "method": "POST" },
    "on_dismiss":   { "url": "https://api.bowerbird.pro/webhooks/payments", "method": "POST" }
  }'
```

## body_template — opt out of the canonical body

Each event handler accepts an optional `body_template` object. When present,
the dispatcher emits **whatever you define** instead of the canonical body —
you can interpolate the following variables with `{{var}}` syntax:

`transaction_id`, `status`, `amount`, `usd_amount`, `currency`,
`currency_provider`, `user_id`, `billing_id`, `provider`, `reference`,
`external_transaction_id`.

When `body_template` is **absent or empty**, the dispatcher falls back to the
[canonical body](./receiving-notifications#canonical-payload). This is the
recommended path for new integrations.
