# Flow: Admin Operations

## Overview

This flow documents the **tenant-admin surface** (`/api/v1/admin/*`) — the endpoints a workspace administrator uses to manage their own tenant: list/invite/update users, edit App settings, rotate the workspace API key, run aggregated reads (businesses, invoices, transactions, domains, jobs, logs), issue short-lived impersonation tokens, and pull tenant-wide metrics.

Every endpoint is gated by the **`app_admin` role**. A user must hold this role for their App before any `/admin/*` call succeeds. Cross-tenant lookups return **404**, never 403, so existence outside the actor's scope is never leaked.

**Use case:** Build a workspace administration panel — invite teammates, audit activity, troubleshoot a customer's data by "viewing as" them, rotate the integration API key after a leak.

**End result:** Full programmatic control of a single workspace, with audit-friendly impersonation and a strict tenant boundary on every read.

---

## Prerequisites

| Requirement | Details |
|---|---|
| App API Key | Application key (`ak-...`) provided when registering your app |
| User API Key | User key (`sk-...`) belonging to an account that holds the `app_admin` role |
| `app_admin` role assigned | The bootstrap admin from `provision_app` already has it; otherwise, an existing admin must grant it via `PATCH /admin/users/{id}` |

### Required headers for all requests

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

:::tip
The `app_admin` role is **tenant-scoped**. It is distinct from the platform-wide `is_superadmin` flag, which gates `/api/v1/platform/*`. A superadmin can additionally call any `/admin/*` endpoint of any tenant (the guard short-circuits for them); a regular `app_admin` only sees their own workspace.
:::

## Step-by-step flow

### Step 0 — Get App Token

**Goal:** Authenticate the application.

**Endpoint:** `POST /api/v1/auth/token`

```bash
APP_TOKEN=$(curl -s -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d "{\"apiKey\":\"$APP_API_KEY\"}" \
  | jq -r '.data.access_token')
```

### Step 1 — Get User Token

**Goal:** Authenticate the admin user.

**Endpoint:** `POST /api/v1/users/token`

```bash
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"apiKey\":\"$USER_API_KEY\"}" \
  | jq -r '.data.access_token')
```

:::note
If the user does NOT hold the `app_admin` role, **every** `/admin/*` request below returns `403 Forbidden` with `"Tenant admin privileges required"`.
:::

---

### Step 2 — List Users

**Goal:** Paginate through the workspace's users, optionally filtered by search term, status, or role name.

**Endpoint:** `GET /api/v1/admin/users`

**Query parameters:**

| Param | Type | Default | Description |
|---|---|---|---|
| `page` | int (1–1000) | 1 | 1-indexed page |
| `page_size` | int (1–200) | 50 | Items per page |
| `search` | string | — | Prefix match against email + username (regex-anchored, escaped) |
| `status` | `active` \| `inactive` | — | Filter by activation state |
| `role` | string | — | Filter by role name (e.g. `app_admin`) |

```bash
curl -s "$BASE_URL/admin/users?page=1&page_size=25&search=alice" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Sample response:**

```json
{
  "success": true,
  "data": {
    "total": 12,
    "page": 1,
    "page_size": 25,
    "items": [
      {
        "id": "507f1f77bcf86cd799439011",
        "email": "alice@acme.example",
        "username": "alice",
        "is_active": true,
        "role_ids": ["507f1f77bcf86cd799439020"],
        "verified_at": "2026-05-12T08:14:02Z",
        "created_at": "2026-05-10T11:22:00Z"
      }
    ]
  },
  "msg": "Users retrieved"
}
```

### Step 3 — Invite a User

**Goal:** Issue a magic-link invite by email. Idempotent on the email; re-invitations simply mint a fresh link.

**Endpoint:** `POST /api/v1/admin/users/invite`

**Rate limit:** 10 invites / hour per caller.

```bash
curl -s -X POST "$BASE_URL/admin/users/invite" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"email":"newcomer@acme.example"}'
```

**Sample response (202 Accepted):**

```json
{
  "success": true,
  "data": { "email": "newcomer@acme.example" },
  "msg": "Invite dispatched"
}
```

:::warning
If the magic-link email cannot be dispatched (e.g. mail provider down) the endpoint returns **502 Bad Gateway** and invalidates the freshly-issued link in the database. Retry the call once the upstream is healthy. No token leaks from the response body, and the orphan link is consumed so it cannot be replayed from logs.
:::

---

### Step 4 — Patch a User

**Goal:** Toggle activation or replace the user's role list.

**Endpoint:** `PATCH /api/v1/admin/users/{user_id}`

**Body fields (all optional, at least one required):**

| Field | Type | Description |
|---|---|---|
| `is_active` | bool | Toggle activation. `false` soft-deactivates the user. |
| `role_ids` | string[] | Replace the role list. Role ids outside the tenant scope are rejected with `400`. |

```bash
curl -s -X PATCH "$BASE_URL/admin/users/$TARGET_USER_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"is_active": false}'
```

**Safeguards:**

- **409 Conflict — "Cannot remove the last active app_admin"** when the patch would leave the tenant with zero active admins.
- **404 Not Found** when the target user is not in the caller's tenant.

---

### Step 5 — Rotate the Workspace API Key

**Goal:** Issue a fresh App `api_key`. Returned **once** — store it immediately.

**Endpoint:** `POST /api/v1/admin/app/keys/rotate`

**Rate limit:** 3 rotations / hour.

```bash
ROTATE_RESP=$(curl -s -X POST "$BASE_URL/admin/app/keys/rotate" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN")

NEW_API_KEY=$(echo "$ROTATE_RESP" | jq -r '.data.api_key')
```

**Sample response (200):**

```json
{
  "success": true,
  "data": {
    "api_key": "ak-NEW-shiny-key-shown-once",
    "rotated_at": "2026-05-20T13:42:00Z"
  },
  "msg": "App api_key rotated"
}
```

:::warning
**The new key invalidates every active integration token immediately.** Update your CI pipelines and the WordPress plugin configuration before the next cron tick fires. The response carries `Cache-Control: no-store` so intermediaries should never cache the body.
:::

---

### Step 6 — Impersonate a User

**Goal:** Issue a 15-minute JWT the admin can use as the target user's `x-user-token`. The original admin id is preserved in the token's `impersonated_by` claim for audit attribution.

**Endpoint:** `POST /api/v1/admin/users/{user_id}/impersonate`

**Rate limit:** 5 impersonations / hour.

```bash
IMP_RESP=$(curl -s -X POST "$BASE_URL/admin/users/$TARGET_USER_ID/impersonate" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"reason":"debug customer issue #4242"}')

IMPERSONATION_TOKEN=$(echo "$IMP_RESP" | jq -r '.data.access_token')
```

**Sample response:**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "token_type": "bearer",
    "expires_in": 900,
    "target_user_id": "507f1f77bcf86cd799439011",
    "impersonated_by": "507f1f77bcf86cd799439099",
    "expires_at": "2026-05-20T13:57:00Z"
  },
  "msg": "Impersonation token issued"
}
```

**Safeguards (all return 4xx):**

| Case | Status | Detail |
|---|---|---|
| Self-impersonation | 400 | `"Cannot impersonate yourself"` |
| Target not in tenant | 404 | `"Resource not found"` |
| Target is the App owner | 403 | `"Cannot impersonate the app owner"` |
| Target is another admin | 403 | `"Cannot impersonate another tenant admin"` |

**Using the token:**

```bash
# Subsequent requests carry the impersonation JWT as x-user-token.
curl -s "$BASE_URL/users/me" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $IMPERSONATION_TOKEN"
```

:::warning
Impersonation tokens are **refused** on a curated set of sensitive endpoints — `/auth/*`, `/auth-user/*`, `/admin/*`, `/platform/*`, `/billing/*`, `/payments/*`, and `/users/me/api-key`. The admin cannot rotate the target's credentials, top up their balance, or nest another impersonation while "viewing as".

The token is also revoked automatically when:
- the **actor** is deactivated, demoted out of `app_admin`, or moved to a different tenant
- the **target** rotates their tokens (logout/revoke bumps their `token_version`)
:::

### Step 7 — Aggregated Reads

**Goal:** Pull tenant-scoped lists across the resource catalogue without leaking cross-tenant data. Each endpoint paginates with `page` + `page_size`.

| Endpoint | Notes |
|---|---|
| `GET /admin/businesses?owner_user_id=…` | Optional filter by owning user |
| `GET /admin/invoices?status=draft|emitted|cancelled` | |
| `GET /admin/transactions?status=created|approved|denied|cancelled|expired` | |
| `GET /admin/domains` | |
| `GET /admin/jobs?status=pending|running|completed|failed|cancelled` | |
| `GET /admin/logs?provider=&success=true|false` | |

```bash
curl -s "$BASE_URL/admin/invoices?page=1&page_size=50&status=emitted" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Sample response:**

```json
{
  "success": true,
  "data": {
    "total": 87,
    "page": 1,
    "page_size": 50,
    "items": [
      {
        "id": "660ab4f9-ce11-4d50-a8d4-7a2c1b30ee10",
        "user_id": "507f1f77bcf86cd799439011",
        "status": "emitted",
        "total": 1234.56,
        "currency": "GTQ",
        "created_at": "2026-05-19T10:30:00Z"
      }
    ]
  },
  "msg": "Invoices retrieved"
}
```

:::note
Response items are projected against a per-resource **field allowlist** server-side. New columns added to the underlying documents do not automatically leak through `/admin/*` — they must be opted into the allowlist explicitly.
:::

### Step 8 — Tenant Metrics

**Goal:** Pull a snapshot of workspace activity for dashboard widgets.

**Endpoint:** `GET /api/v1/admin/metrics`

```bash
curl -s "$BASE_URL/admin/metrics" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Sample response:**

```json
{
  "success": true,
  "data": {
    "users_total": 18,
    "users_active": 16,
    "businesses_total": 4,
    "invoices_mtd": 87,
    "credits_consumed_mtd": 412.5,
    "jobs_running": 2,
    "jobs_failed_24h": 0,
    "top_users": [
      { "user_id": "507f...", "email": "alice@acme.example", "calls": 1240 }
    ],
    "generated_at": "2026-05-20T13:42:01Z"
  },
  "msg": "Metrics retrieved"

```

:::note
The response is **cached in-memory for 60 seconds per tenant**. Successive calls inside the window return the same snapshot — useful for dashboards refreshing on a short interval.
:::

## Error handling

| HTTP | When |
|---|---|
| **401** | Missing / expired / revoked token |
| **403** | Authenticated user does not hold `app_admin`, or target is owner / another admin (impersonation) |
| **404** | Resource not in caller's tenant (NEVER 403 — existence is not leaked) |
| **409** | Owner deletion, last-admin demotion, slug collision |
| **429** | Rate-limited (`10/h` invites, `5/h` impersonations, `3/h` rotations, `30/min` writes, `60/min` reads) |
| **502** | Magic-link email dispatch failed during invite |

Every endpoint wraps responses in the standard envelope:

```json
{ "success": true|false, "data": …, "msg": "human-readable status" }
```

Error responses set `data: null` and place the description in `msg`.

## Best practices

- **Audit impersonation.** Tag your access logs with the `impersonated_by` claim from the JWT payload so every action carries an attributable actor even when the token's subject is the target user.
- **Confirm before rotating.** Surface a confirmation modal before calling `POST /admin/app/keys/rotate` — the new key is shown once and the old one is invalidated immediately.
- **Don't poll metrics faster than once a minute.** The endpoint already caches for 60s, so faster polling returns the same payload and wastes the rate budget you might need elsewhere.
- **Detect last-admin lock-outs client-side.** Before sending `PATCH` with `is_active: false` on a known admin, count the workspace's active admins (via `?role=app_admin&status=active`) and disable the action when the count is 1. The backend will also refuse with 409, but UX feedback is better surfaced upstream.
- **Refuse impersonation when your local session token is unknown.** A client that stashes an empty string as the "original token" cannot restore the admin's session on exit. Read the current access token from your storage layer before issuing the impersonation; abort if `null`.

## End-to-end example

```bash
#!/usr/bin/env bash
set -euo pipefail

BASE_URL="https://api.1platform.pro/api/v1"
APP_API_KEY="ak-your-app-api-key"
USER_API_KEY="sk-admin-abc123"

# 0. App token
APP_TOKEN=$(curl -s -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d "{\"apiKey\":\"$APP_API_KEY\"}" | jq -r '.data.access_token')

# 1. User token
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"apiKey\":\"$USER_API_KEY\"}" | jq -r '.data.access_token')

AUTH=(-H "Authorization: Bearer $APP_TOKEN" -H "x-user-token: $USER_TOKEN")

# 2. List users
curl -s "$BASE_URL/admin/users?page=1&page_size=10" "${AUTH[@]}" | jq '.data.items[].email'

# 3. Invite a new teammate
curl -s -X POST "$BASE_URL/admin/users/invite" \
  "${AUTH[@]}" -H "Content-Type: application/json" \
  -d '{"email":"newcomer@acme.example"}' | jq

# 4. Pull a metrics snapshot
curl -s "$BASE_URL/admin/metrics" "${AUTH[@]}" | jq '.data | {users_total, users_active, jobs_running}'

# 5. Rotate the workspace key (commented — this invalidates active tokens)
# curl -s -X POST "$BASE_URL/admin/app/keys/rotate" "${AUTH[@]}" | jq '.data.api_key'
```
