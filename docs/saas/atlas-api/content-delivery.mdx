# Entrega de contenido

Atlas nunca entrega un enlace permanente al archivo subyacente. En su lugar, un miembro autenticado
solicita un **ticket de reproducción** — una credencial de corta duración, verificada por entitlement, que apunta al
asset. Dependiendo del [arquetipo](/docs/saas/atlas-api/tenant-resolution) del tenant, el ticket es
una **URL firmada** (archivos descargables: ebooks, audio) o un **envoltorio de streaming HLS** (video,
live, TV) que porta un token de DRM.

## Generar un ticket de reproducción

```bash
curl -s "$ATLAS_BASE/delivery/items/<CONTENT_ID>/playback?format=epub" \
  -H "X-Tenant-Slug: $TENANT_SLUG" \
  -H "Authorization: Bearer $ATLAS_JWT"
```

```json
{
  "content_id": "a3b4…",
  "delivery_kind": "signed_url",
  "url": "https://storage.example/…?signature=…&expires=…",
  "expires_in_seconds": 600,
  "drm_token": null,
  "renderable": true,
  "render_format": "epub"
}
```

### Campos de la respuesta

| Campo | Descripción |
|---|---|
| `content_id` | El ítem para el que es este ticket. |
| `delivery_kind` | `signed_url` (descarga de archivo) o `hls` (streaming). |
| `url` | La URL del asset — un enlace de descarga firmado, o una URL de manifiesto HLS. |
| `expires_in_seconds` | Vida útil del ticket (actualmente ~600s / 10 minutos). Vuelve a solicitarlo tras la expiración. |
| `drm_token` | Presente en envoltorios HLS; ligado a (content, member, expiry). `null` para descargas con URL firmada. |
| `renderable` | `false` solo cuando un formato aún no tiene una representación renderizable — el cliente debe degradar a descargar/abrir externamente en lugar de abrir un lector in-app. |
| `render_format` | El formato del motor con el que el cliente debe renderizar (p. ej., `epub`). |

### La pista `format`

La consulta opcional `?format=` selecciona qué asset subido entregar — `pdf`, `epub`, `mobi`,
`mp3`, `m4a`, etc. Cuando se omite, la estrategia elige el primer formato subido de los
metadatos del ítem, o el formato por defecto de su estrategia.

```bash
# Descarga de audio
curl -s "$ATLAS_BASE/delivery/items/<CONTENT_ID>/playback?format=mp3" \
  -H "X-Tenant-Slug: $TENANT_SLUG" -H "Authorization: Bearer $ATLAS_JWT"
```

Un formato que no esté en la lista de permitidos del tenant devuelve `403 { "code": "ASSET_FORMAT_NOT_ALLOWED" }`.

## Estrategia de entrega por arquetipo

La estrategia de entrega se resuelve a partir del arquetipo del tenant — no la eliges por solicitud:

| Arquetipos | `delivery_kind` | Forma |
|---|---|---|
| `library`, `podcast`, `education` | `signed_url` | Una URL firmada y con expiración al archivo. El cliente lo descarga o lo abre. |
| `streaming`, `live`, `tv`, `events` | `hls` | Una URL de manifiesto HLS más un `drm_token` ligado al miembro y a la expiración. |

## Control de acceso

Antes de generar un ticket, Atlas hace cumplir el acceso:

1. **Los operadores se saltan la verificación.** Un llamador con rol `admin` u `operator` se salta la verificación de entitlement
   y las guardas de formato/cuota (son dueños del catálogo que subieron). Cada salto se escribe en el log de auditoría.
2. **Entitlement activo.** Un miembro regular debe tener un entitlement activo que cubra el ítem.
3. **Auto-grant perezoso.** En tenants de nivel gratuito, si el miembro aún no tiene entitlement, Atlas intenta un
   grant en línea y vuelve a verificar — cubriendo el caso en que el grant al iniciar sesión no se concretó.
4. **Denegado.** De lo contrario, la solicitud falla con el muro de suscripción:

```json
{ "code": "ENTITLEMENT_DENIED" }
```

(HTTP `403`.) Consulta [Entitlements](/docs/saas/atlas-api/entitlements) para saber cómo se concede el acceso.

## Flujo del cliente

```text
GET /public/items/{slug}                         → descubre el ítem (anónimo)
GET /delivery/items/{content_id}/playback        → genera un ticket (JWT requerido)
   ├─ delivery_kind = signed_url  → GET url       → descarga / abre el archivo
   └─ delivery_kind = hls         → play url       → el reproductor HLS usa drm_token
(el ticket expira)                                → vuelve a solicitar el ticket
```

Los tickets son intencionalmente de corta duración. Trata la `url` como efímera: obtenla justo antes de la reproducción,
no la caches ni la compartas, y vuelve a generarla cuando transcurra `expires_in_seconds`.
