# Flow: AI Agents

## Overview

This flow documents the complete lifecycle of AI agents through the 1Platform API. Agents are autonomous LLM-powered workers that execute tasks, call tools, and request human approval when needed.

**Use case:** Automate recurring business tasks — content publishing, invoice review, email drafting — by creating agents from templates or custom configurations, scheduling or triggering runs, and consuming pending actions that require human decisions.

**End result:** A configured, active agent that executes tasks autonomously with full observability into runs, metrics, and human-in-the-loop actions.

## Data required to create an agent

### Agent configuration

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | Yes | Human-readable agent name (max 200 chars) |
| `description` | string | Yes | What the agent does (max 1000 chars) |
| `system_prompt` | string | Yes | System prompt for the LLM (max 10000 chars) |
| `model` | string | Yes | LLM model: `"claude-sonnet-4-6"` or `"claude-haiku-4-5-20251001"` |
| `tools` | array | No | Tools available to the agent (see Tool configuration below) |
| `max_iterations` | integer | No | Maximum agentic loop iterations (1-50, default: `10`) |
| `schedule` | string | No | Cron expression for scheduled runs (omit to disable) |
| `webhook_url` | string | No | Webhook URL for run result notifications |
| `is_active` | boolean | No | Whether the agent is immediately active (default: `true`) |
| `max_concurrent_runs` | integer | No | Maximum simultaneous runs (1-3, default: `1`) |
| `template_slug` | string | No | Catalog template slug if created from a template |

### Tool configuration

Each tool in the `tools` array has:

| Field | Type | Required | Description |
|---|---|---|---|
| `type` | string | Yes | `"internal"` (platform tool) or `"external"` (HTTP call) |
| `definition.name` | string | Yes | Unique tool name within the agent |
| `definition.description` | string | Yes | Natural-language description of what the tool does |
| `definition.input_schema` | object | Yes | JSON Schema describing tool inputs |
| `execution_config` | object | No | Required when `type` is `"external"` |
| `execution_config.base_url` | string | No | Base URL of the external tool endpoint |
| `execution_config.encrypted_headers` | string | No | Fernet-encrypted JSON string of HTTP headers |
| `execution_config.timeout` | integer | No | Request timeout in seconds (max 30, default: `30`) |

## Step-by-step flow

### Step 0 — Get App Token

**Goal:** Authenticate the application and obtain the app JWT token.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** None (this is the entry point)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Response (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Save `data.access_token` as `$APP_TOKEN`. Expires in 30 minutes.
:::

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 401 | Invalid API key or inactive app | Verify that `apiKey` is correct and the app is active |
| 422 | Missing or malformed `apiKey` field | Send `{"apiKey": "..."}` in the body |
| 429 | Rate limit exceeded (10 req/min) | Wait and retry |

### Step 4 — Start Wizard Session

**Goal:** Begin an interactive wizard session that guides you through agent configuration. The wizard asks questions from the template and generates a complete agent configuration.

**Endpoint:** `POST /api/v1/agents/wizard/start`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
# Start from a template
curl -X POST "$BASE_URL/agents/wizard/start" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "template_slug": "content-publisher"
  }'
```

You can also start without a template by providing an `initial_message`:

```bash
# Start from scratch
curl -X POST "$BASE_URL/agents/wizard/start" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "initial_message": "I need an agent that monitors my website SEO rankings daily and alerts me when a keyword drops out of the top 10"
  }'
```

:::note
At least one of `template_slug` or `initial_message` must be provided.
:::

**Response (201):**

```json
{
  "success": true,
  "data": {
    "id": "60a1b2c3d4e5f6789abcde01",
    "template_slug": "content-publisher",
    "status": "in_progress",
    "messages": [
      {
        "role": "assistant",
        "content": "Great! Let's set up your Content Publisher agent. First, what topics should the agent cover?",
        "timestamp": "2026-03-21T15:00:00Z"
      }
    ],
    "generated_config": null,
    "expires_at": "2026-03-21T16:00:00Z"
  },
  "msg": "Wizard session created"
}
```

:::note
Save `data.id` as `$SESSION_ID`. The session expires at `expires_at` — complete the wizard before then.
:::

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 400 | Validation error (neither `template_slug` nor `initial_message` provided) | Provide at least one |
| 401 | Authentication required | Regenerate tokens |
| 404 | Template not found | Verify the template slug exists in the catalog |
| 429 | Rate limit exceeded | Wait and retry |

```bash
curl -X POST "$BASE_URL/agents/wizard/$SESSION_ID/respond" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "message": "Focus on SEO and content marketing topics. Publish every Monday at 9 AM."
  }'
```

**Response (200):**

```json
{
  "success": true,
  "data": {
    "id": "60a1b2c3d4e5f6789abcde01",
    "template_slug": "content-publisher",
```

### Step 9 — Trigger an Agent Run

**Goal:** Manually trigger an agent run with an optional input message.

**Endpoint:** `POST /api/v1/agents/{agent_id}/run`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/agents/$AGENT_ID/run" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "input_message": "Review all invoices created today and flag any over Q50,000"
  }'
```

**Response (202):**

```json
{
  "success": true,
  "data": {
    "id": "60d3e4f5a6b7c8901def2045",
    "agent_id": "60c2d3e4f5a6b7890cdef103",
    "status": "pending",
    "trigger": "manual",
    "input_message": "Review all invoices created today and flag any over Q50,000",
    "result": null,
    "error": null,
    "total_tokens": 0,
    "worker_id": null,
    "started_at": null,
    "finished_at": null,
    "created_at": "2026-03-21T15:15:00Z"
  },
  "msg": "Run triggered \u2014 executing in background"

```

:::note
Save `data.id` as `$RUN_ID`. The run starts in `"pending"` status and transitions to `"running"`, then `"completed"`, `"failed"`, or `"stopped"`.

**Common errors:**

| Code | Cause | Solution |
|---|---|---|

### Step 11 — List Pending Actions

**Goal:** Retrieve human-in-the-loop actions generated by agent runs that require human decisions.

**Endpoint:** `GET /api/v1/agent-actions`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
# List all pending actions
curl -X GET "$BASE_URL/agent-actions?status=pending" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"

# Filter by type and agent
curl -X GET "$BASE_URL/agent-actions?type=review_invoice&agent_id=$AGENT_ID&status=pending" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

| Parameter | Type | Default | Description |
|---|---|---|---|
| `type` | string | — | Filter by action type: `publish_content`, `review_invoice`, `approve_issue`, `send_email` |
| `status` | string | — | Filter by status: `pending` or `consumed` |
| `agent_id` | string | — | Filter by agent ID |
| `skip` | integer | `0` | Number of results to skip |
| `limit` | integer | `20` | Maximum results to return |

### Step 12 — Consume an Action

**Goal:** Resolve a pending human-in-the-loop action by providing your decision.

**Endpoint:** `PATCH /api/v1/agent-actions/{action_id}/consume`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X PATCH "$BASE_URL/agent-actions/$ACTION_ID/consume" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "result": {
      "approved": true,
      "notes": "Verified with finance team. Amount is correct for the annual contract."
    },
    "consumed_by": "maria@example.com"
  }'
```

**Request body:**

## Full end-to-end example

Complete flow: create an agent via wizard, trigger a run, and consume an action.

```bash
# === VARIABLES ===
BASE_URL="https://api.1platform.pro/api/v1"

# === STEP 0: Get App Token ===
APP_TOKEN=$(curl -s -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "ak-prod-app-key-2026"}' \
  | jq -r '.data.access_token')

echo "APP_TOKEN: $APP_TOKEN"

# === STEP 1: Get User Token ===
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "sk-user-maria-2026"}' \
  | jq -r '.data.access_token')

echo "USER_TOKEN: $USER_TOKEN"

# === STEP 2: Browse Catalog ===
curl -s -X GET "$BASE_URL/agents/catalog" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq '.data.templates[] | {slug, name, category}'

# === STEP 3: Create Agent Directly ===
AGENT_ID=$(curl -s -X POST "$BASE_URL/agents" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Invoice Anomaly Detector",
    "description": "Reviews invoices and flags high amounts for human approval",
    "system_prompt": "You are a financial review agent. Examine invoices and flag any with amounts exceeding Q50,000.",
    "model": "claude-haiku-4-5-20251001",
    "max_iterations": 10,
    "is_active": true
  }' | jq -r '.data.id')

echo "AGENT_ID: $AGENT_ID"

# === STEP 4: Trigger a Run ===
RUN_ID=$(curl -s -X POST "$BASE_URL/agents/$AGENT_ID/run" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "input_message": "Review all invoices from today"
  }' | jq -r '.data.id')

echo "RUN_ID: $RUN_ID"

# === STEP 5: Poll Run Status ===
sleep 5
RUN_STATUS=$(curl -s -X GET "$BASE_URL/agents/$AGENT_ID/runs/$RUN_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq -r '.data.status')

echo "Run status: $RUN_STATUS"

# === STEP 6: List Pending Actions ===
ACTIONS=$(curl -s -X GET "$BASE_URL/agent-actions?agent_id=$AGENT_ID&status=pending" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN")

echo "Pending actions: $(echo $ACTIONS | jq '.data.total')"

ACTION_ID=$(echo $ACTIONS | jq -r '.data.actions[0].id')

# === STEP 7: Consume Action ===
if [ "$ACTION_ID" != "null" ]; then
  curl -s -X PATCH "$BASE_URL/agent-actions/$ACTION_ID/consume" \
    -H "Authorization: Bearer $APP_TOKEN" \
    -H "x-user-token: $USER_TOKEN" \
    -H "Content-Type: application/json" \
    -d '{
      "result": {"approved": true, "notes": "Verified and approved"},
      "consumed_by": "maria@example.com"
    }' | jq .
fi

# === STEP 8: Check Metrics ===
curl -s -X GET "$BASE_URL/agents/metrics" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .
```

## Idempotency and best practices

### Idempotency
- Agent creation is **not idempotent** — calling `POST /agents` twice with the same data creates two agents. Check `GET /agents` before retrying.
- Run triggers are **not idempotent** — each `POST /agents/{id}/run` creates a new run. Check `GET /agents/{id}/runs` before retrying.
- Action consumption is idempotent in effect — consuming an already-consumed action returns a **409** error.

### Retries
- If you receive **429** (rate limit), wait the indicated time and retry.
- If you receive **500** or a **timeout**, verify the resource state before retrying to avoid duplicates.
- For **401** (expired token), regenerate the token and retry the original request.
- For **409** on run trigger (concurrent limit), wait for current runs to finish.
- For **503** (capacity full), wait for global capacity to free up.

### Timeouts
- Catalog and CRUD endpoints typically respond within **2-5 seconds**.
- Wizard responses may take **5-15 seconds** as the LLM processes the conversation.
- Run triggers return immediately (**202**) — the actual execution is asynchronous.
- Agent runs can take **seconds to minutes** depending on the task complexity and number of iterations.

### Tokens
- Both tokens (app and user) expire in **30 minutes** (1800 seconds). Implement automatic renewal before expiration.
- Both tokens are required on all agent endpoints.

### Important validations
- `model` must be one of: `"claude-sonnet-4-6"` or `"claude-haiku-4-5-20251001"`.
- `max_iterations` must be between 1 and 50.
- `max_concurrent_runs` must be between 1 and 3.
- Wizard sessions expire — complete them before `expires_at`.
- External tools require an `execution_config` with at least a `base_url`.
- `schedule` follows standard cron expression syntax.
