# Flujo: Agentes de IA

## Vista general

Este flujo documenta el ciclo de vida completo de los agentes de IA mediante la 1Platform API. Los agentes son trabajadores autónomos impulsados por LLM que ejecutan tareas, llaman a herramientas y solicitan aprobación humana cuando es necesario.

**Caso de uso:** Automatizar tareas de negocio recurrentes — publicación de contenido, revisión de facturas, redacción de correos — creando agentes a partir de plantillas o configuraciones personalizadas, programando o disparando ejecuciones, y consumiendo acciones pendientes que requieren decisiones humanas.

**Resultado final:** Un agente configurado y activo que ejecuta tareas de forma autónoma con observabilidad completa sobre ejecuciones, métricas y acciones con intervención humana.

## Requisitos previos

| Requisito | Detalles |
|---|---|
| App API Key | Clave de aplicación (`ak-...`) proporcionada al registrar tu app |
| User API Key | Clave de usuario (`sk-...`) obtenida al crear el usuario |

### Headers requeridos para todas las solicitudes

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

## Variables usadas en los ejemplos

```bash
BASE_URL="https://api.1platform.pro/api/v1"
APP_API_KEY="ak-your-app-api-key"
USER_API_KEY="sk-user-abc123"
APP_TOKEN=""        # Obtenido en el Paso 0

AGENT_ID=""         # Obtenido en el Paso 6 (o el Paso 8 para creación directa)

```

## Datos requeridos para crear un agente

### Configuración del agente

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `name` | string | Sí | Nombre legible del agente (máx. 200 caracteres) |
| `description` | string | Sí | Qué hace el agente (máx. 1000 caracteres) |
| `system_prompt` | string | Sí | Prompt de sistema para el LLM (máx. 10000 caracteres) |
| `model` | string | Sí | Modelo de LLM: `"claude-sonnet-4-6"` o `"claude-haiku-4-5-20251001"` |
| `tools` | array | No | Herramientas disponibles para el agente (ver Configuración de herramientas más abajo) |
| `max_iterations` | integer | No | Iteraciones máximas del bucle agéntico (1-50, por defecto: `10`) |
| `schedule` | string | No | Expresión cron para ejecuciones programadas (omitir para deshabilitar) |
| `webhook_url` | string | No | URL del webhook para notificaciones de resultados de ejecución |
| `is_active` | boolean | No | Si el agente está activo de inmediato (por defecto: `true`) |
| `max_concurrent_runs` | integer | No | Ejecuciones simultáneas máximas (1-3, por defecto: `1`) |
| `template_slug` | string | No | Slug de la plantilla del catálogo si se creó a partir de una plantilla |

### Configuración de herramientas

Cada herramienta en el array `tools` tiene:

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `type` | string | Sí | `"internal"` (herramienta de la plataforma) o `"external"` (llamada HTTP) |
| `definition.name` | string | Sí | Nombre único de la herramienta dentro del agente |
| `definition.description` | string | Sí | Descripción en lenguaje natural de lo que hace la herramienta |
| `definition.input_schema` | object | Sí | JSON Schema que describe las entradas de la herramienta |
| `execution_config` | object | No | Requerido cuando `type` es `"external"` |
| `execution_config.base_url` | string | No | URL base del endpoint de la herramienta externa |
| `execution_config.encrypted_headers` | string | No | Cadena JSON de headers HTTP cifrada con Fernet |
| `execution_config.timeout` | integer | No | Timeout de la solicitud en segundos (máx. 30, por defecto: `30`) |

## Flujo paso a paso

### Paso 0 — Obtener el token de aplicación

**Objetivo:** Autenticar la aplicación y obtener el token JWT de aplicación.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** Ninguna (este es el punto de entrada)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$APP_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key inválida o app inactiva | Verifica que `apiKey` sea correcta y que la app esté activa |
| 422 | Campo `apiKey` ausente o mal formado | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido (10 req/min) | Espera y reintenta |

### Paso 1 — Obtener el token de usuario

**Objetivo:** Autenticar al usuario y obtener el token JWT de usuario.

**Endpoint:** `POST /api/v1/users/token`

**Auth:** `Authorization: Bearer $APP_TOKEN` (solo token de aplicación, no se requiere token de usuario)

```bash
curl -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$USER_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "User token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$USER_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key de usuario inválida o usuario inactivo | Verifica la `apiKey` del usuario |
| 422 | Campo `apiKey` ausente | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido | Espera y reintenta |

:::tip
Si aún no tienes un usuario, crea uno con `POST /api/v1/users` enviando `{"email": "user@example.com"}`. La respuesta incluirá el `api_key` del usuario.
:::

### Paso 4 — Iniciar una sesión del asistente

**Objetivo:** Comenzar una sesión interactiva del asistente que te guía a través de la configuración del agente. El asistente hace preguntas de la plantilla y genera una configuración de agente completa.

**Endpoint:** `POST /api/v1/agents/wizard/start`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
# Iniciar a partir de una plantilla
curl -X POST "$BASE_URL/agents/wizard/start" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "template_slug": "content-publisher"
  }'
```

También puedes iniciar sin una plantilla proporcionando un `initial_message`:

```bash
# Iniciar desde cero
curl -X POST "$BASE_URL/agents/wizard/start" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "initial_message": "I need an agent that monitors my website SEO rankings daily and alerts me when a keyword drops out of the top 10"
  }'
```

:::note
Se debe proporcionar al menos uno de `template_slug` o `initial_message`.
:::

**Respuesta (201):**

```json
{
  "success": true,
  "data": {
    "id": "60a1b2c3d4e5f6789abcde01",
    "template_slug": "content-publisher",
    "status": "in_progress",
    "messages": [
      {
        "role": "assistant",
        "content": "Great! Let's set up your Content Publisher agent. First, what topics should the agent cover?",
        "timestamp": "2026-03-21T15:00:00Z"
      }
    ],
    "generated_config": null,
    "expires_at": "2026-03-21T16:00:00Z"
  },
  "msg": "Wizard session created"
}
```

:::note
Guarda `data.id` como `$SESSION_ID`. La sesión expira en `expires_at` — completa el asistente antes de entonces.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 400 | Error de validación (no se proporcionó ni `template_slug` ni `initial_message`) | Proporciona al menos uno |
| 401 | Autenticación requerida | Regenera los tokens |
| 404 | Plantilla no encontrada | Verifica que el slug de la plantilla exista en el catálogo |
| 429 | Límite de tasa excedido | Espera y reintenta |

### Paso 5 — Responder al asistente

**Objetivo:** Responder las preguntas del asistente para refinar la configuración del agente. Repite este paso por cada pregunta hasta que el estado del asistente cambie a `"completed"`.

**Endpoint:** `POST /api/v1/agents/wizard/{session_id}/respond`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/agents/wizard/$SESSION_ID/respond" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "message": "Focus on SEO and content marketing topics. Publish every Monday at 9 AM."
  }'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "id": "60a1b2c3d4e5f6789abcde01",
    "template_slug": "content-publisher",
    "status": "in_progress",
    "messages": [
      {
        "role": "assistant",
        "content": "Great! Let's set up your Content Publisher agent. First, what topics should the agent cover?",
        "timestamp": "2026-03-21T15:00:00Z"
      },
      {
        "role": "user",
        "content": "Focus on SEO and content marketing topics. Publish every Monday at 9 AM.",
        "timestamp": "2026-03-21T15:01:00Z"
      },
      {
        "role": "assistant",
        "content": "Got it! Which website should the agent publish to?",
        "timestamp": "2026-03-21T15:01:01Z"
      }
    ],
    "generated_config": null,
    "expires_at": "2026-03-21T16:00:00Z"
  },
  "msg": "Wizard response processed"
}
```

:::tip
Cuando `status` cambie a `"completed"`, el campo `generated_config` contendrá la configuración completa del agente. Entonces puedes confirmar el asistente para crear el agente (Paso 6).
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | Autenticación requerida | Regenera los tokens |
| 404 | Sesión no encontrada | Verifica el `session_id` |
| 410 | Sesión expirada | Inicia una nueva sesión del asistente |
| 429 | Límite de tasa excedido | Espera y reintenta |

### Paso 9 — Disparar una ejecución del agente

**Objetivo:** Disparar manualmente una ejecución del agente con un mensaje de entrada opcional.

**Endpoint:** `POST /api/v1/agents/{agent_id}/run`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/agents/$AGENT_ID/run" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "input_message": "Review all invoices created today and flag any over Q50,000"
  }'
```

**Respuesta (202):**

```json
{
  "success": true,
  "data": {
    "id": "60d3e4f5a6b7c8901def2045",
    "agent_id": "60c2d3e4f5a6b7890cdef103",
    "status": "pending",
    "trigger": "manual",
    "input_message": "Review all invoices created today and flag any over Q50,000",
    "result": null,
    "error": null,
    "total_tokens": 0,
    "worker_id": null,
    "started_at": null,
    "finished_at": null,
    "created_at": "2026-03-21T15:15:00Z"
  },
  "msg": "Run triggered — executing in background"
}
```

:::note
Guarda `data.id` como `$RUN_ID`. La ejecución comienza en estado `"pending"` y transiciona a `"running"`, luego a `"completed"`, `"failed"` o `"stopped"`.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | Autenticación requerida | Regenera los tokens |

## Ejemplo completo de extremo a extremo

Flujo completo: crear un agente mediante el asistente, disparar una ejecución y consumir una acción.

```bash
# === VARIABLES ===
BASE_URL="https://api.1platform.pro/api/v1"

# === PASO 0: Obtener el token de aplicación ===
APP_TOKEN=$(curl -s -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "ak-prod-app-key-2026"}' \
  | jq -r '.data.access_token')

echo "APP_TOKEN: $APP_TOKEN"

# === PASO 1: Obtener el token de usuario ===
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "sk-user-maria-2026"}' \
  | jq -r '.data.access_token')

echo "USER_TOKEN: $USER_TOKEN"

# === PASO 2: Explorar el catálogo ===
curl -s -X GET "$BASE_URL/agents/catalog" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq '.data.templates[] | {slug, name, category}'

# === PASO 3: Crear un agente directamente ===
AGENT_ID=$(curl -s -X POST "$BASE_URL/agents" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Invoice Anomaly Detector",
    "description": "Reviews invoices and flags high amounts for human approval",
    "system_prompt": "You are a financial review agent. Examine invoices and flag any with amounts exceeding Q50,000.",
    "model": "claude-haiku-4-5-20251001",
    "max_iterations": 10,
    "is_active": true
  }' | jq -r '.data.id')

echo "AGENT_ID: $AGENT_ID"

# === PASO 4: Disparar una ejecución ===
RUN_ID=$(curl -s -X POST "$BASE_URL/agents/$AGENT_ID/run" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "input_message": "Review all invoices from today"
  }' | jq -r '.data.id')

echo "RUN_ID: $RUN_ID"

# === PASO 5: Sondear el estado de la ejecución ===
sleep 5
RUN_STATUS=$(curl -s -X GET "$BASE_URL/agents/$AGENT_ID/runs/$RUN_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq -r '.data.status')

echo "Run status: $RUN_STATUS"

# === PASO 6: Listar acciones pendientes ===
ACTIONS=$(curl -s -X GET "$BASE_URL/agent-actions?agent_id=$AGENT_ID&status=pending" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN")

echo "Pending actions: $(echo $ACTIONS | jq '.data.total')"

ACTION_ID=$(echo $ACTIONS | jq -r '.data.actions[0].id')

# === PASO 7: Consumir una acción ===
if [ "$ACTION_ID" != "null" ]; then
  curl -s -X PATCH "$BASE_URL/agent-actions/$ACTION_ID/consume" \
    -H "Authorization: Bearer $APP_TOKEN" \
    -H "x-user-token: $USER_TOKEN" \
    -H "Content-Type: application/json" \
    -d '{
      "result": {"approved": true, "notes": "Verified and approved"},
      "consumed_by": "maria@example.com"
    }' | jq .
fi

# === PASO 8: Comprobar métricas ===
curl -s -X GET "$BASE_URL/agents/metrics" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .
```

## Idempotencia y buenas prácticas

### Idempotencia
- La creación de agentes **no es idempotente** — llamar a `POST /agents` dos veces con los mismos datos crea dos agentes. Comprueba `GET /agents` antes de reintentar.
- Los disparadores de ejecución **no son idempotentes** — cada `POST /agents/{id}/run` crea una nueva ejecución. Comprueba `GET /agents/{id}/runs` antes de reintentar.
- El consumo de acciones es idempotente en efecto — consumir una acción ya consumida devuelve un error **409**.

### Reintentos
- Si recibes **429** (límite de tasa), espera el tiempo indicado y reintenta.
- Si recibes **500** o un **timeout**, verifica el estado del recurso antes de reintentar para evitar duplicados.
- Para **401** (token expirado), regenera el token y reintenta la solicitud original.
- Para **409** al disparar una ejecución (límite concurrente), espera a que terminen las ejecuciones actuales.
- Para **503** (capacidad llena), espera a que se libere la capacidad global.

### Timeouts
- Los endpoints de catálogo y CRUD normalmente responden en **2-5 segundos**.
- Las respuestas del asistente pueden tardar **5-15 segundos** mientras el LLM procesa la conversación.
- Los disparadores de ejecución devuelven de inmediato (**202**) — la ejecución real es asíncrona.
- Las ejecuciones del agente pueden tardar **de segundos a minutos** según la complejidad de la tarea y el número de iteraciones.

### Tokens
- Ambos tokens (de aplicación y de usuario) expiran en **30 minutos** (1800 segundos). Implementa una renovación automática antes de la expiración.
- Ambos tokens son requeridos en todos los endpoints de agentes.

### Validaciones importantes
- `model` debe ser uno de: `"claude-sonnet-4-6"` o `"claude-haiku-4-5-20251001"`.
- `max_iterations` debe estar entre 1 y 50.
- `max_concurrent_runs` debe estar entre 1 y 3.
- Las sesiones del asistente expiran — complétalas antes de `expires_at`.
- Las herramientas externas requieren un `execution_config` con al menos un `base_url`.
- `schedule` sigue la sintaxis estándar de expresiones cron.
