# Flujo: Google AdSense

## Vista general

Este flujo documenta cómo integrar Google AdSense con tus sitios web a través de la 1Platform API. Cubre la autorización OAuth, la conexión de cuentas de AdSense, la consulta de ingresos (vista general, por página, reportes personalizados), la gestión de sitios, el monitoreo de alertas y problemas de políticas, y la desconexión o reparación de conexiones.

**Caso de uso:** Conectar los sitios web de tus usuarios a Google AdSense, monitorear los ingresos y el rendimiento de los anuncios, hacer seguimiento de problemas de políticas y alertas, y gestionar todo el ciclo de vida de AdSense, todo a través de una API unificada.

**Resultado final:** Una cuenta de Google AdSense completamente conectada con acceso a datos de ingresos, gestión de sitios, monitoreo de alertas y seguimiento de problemas de políticas.

---

## Requisitos previos

| Requisito | Detalles |
|---|---|
| App API Key | Clave de aplicación (`ak-...`) proporcionada al registrar tu app |
| User API Key | Clave de usuario (`sk-...`) obtenida al crear el usuario |
| Sitio web registrado | Debe registrarse un sitio web mediante el flujo de Gestión de Sitios Web |
| Cuenta de Google AdSense | Se requiere una cuenta activa de Google AdSense |

### Headers requeridos para todas las solicitudes

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

## Variables usadas en los ejemplos

```bash
BASE_URL="https://api.1platform.pro/api/v1"
APP_API_KEY="ak-your-app-api-key"
USER_API_KEY="sk-user-abc123"
APP_TOKEN=""      # Obtenido en el Paso 0
USER_TOKEN=""     # Obtenido en el Paso 1
WEBSITE_ID=""     # ID del sitio web al que conectar AdSense
```

## Flujo paso a paso

### Paso 0 — Obtener el token de aplicación

**Objetivo:** Autenticar la aplicación y obtener el token JWT de la app.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** Ninguna (este es el punto de entrada)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$APP_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key inválida o app inactiva | Verifica que `apiKey` sea correcto y que la app esté activa |
| 422 | Campo `apiKey` ausente o mal formado | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido (10 req/min) | Espera y reintenta |

### Paso 1 — Obtener el token de usuario

**Objetivo:** Autenticar al usuario y obtener el token JWT de usuario.

**Endpoint:** `POST /api/v1/users/token`

**Auth:** `Authorization: Bearer $APP_TOKEN` (solo token de aplicación, no se requiere token de usuario)

```bash
curl -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$USER_API_KEY'"}'
```

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "User token generated successfully"
}
```

:::note
Guarda `data.access_token` como `$USER_TOKEN`. Expira en 30 minutos.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | API key de usuario inválida o usuario inactivo | Verifica el `apiKey` del usuario |
| 422 | Campo `apiKey` ausente | Envía `{"apiKey": "..."}` en el cuerpo |
| 429 | Límite de tasa excedido | Espera y reintenta |

:::tip
Si aún no tienes un usuario, crea uno con `POST /api/v1/users` enviando `{"email": "user@example.com"}`. La respuesta incluirá el `api_key` del usuario.
:::

### Paso 2 — Iniciar la autorización OAuth

**Objetivo:** Iniciar el flujo OAuth de Google para AdSense. La API devuelve una URL de autorización que el usuario debe visitar en su navegador para conceder el acceso.

**Endpoint:** `GET /api/v1/adsense/oauth/authorize`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X GET "$BASE_URL/adsense/oauth/authorize?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Parámetros de consulta:**

| Parámetro | Tipo | Requerido | Descripción |
|---|---|---|---|
| `website_id` | string | Sí | ID del sitio web para el que se autoriza AdSense |

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "authorization_url": "https://accounts.google.com/o/oauth2/v2/auth?client_id=...&redirect_uri=...&scope=https://www.googleapis.com/auth/adsense.readonly&state=507f1f77bcf86cd799439011",
    "state": "507f1f77bcf86cd799439011"
  },
  "msg": "Authorization URL generated successfully"
}
```

:::warning
El usuario debe abrir la `authorization_url` en su navegador, iniciar sesión con su cuenta de Google y conceder los permisos de AdSense. El callback OAuth lo gestiona automáticamente 1Platform. Tras la autorización, usa el Paso 3 para verificar el estado de OAuth antes de conectar.
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 401 | Token de autenticación ausente o inválido | Regenera los tokens (Pasos 0 y 1) |
| 422 | Error de validación — `website_id` ausente o inválido | Proporciona un parámetro de consulta `website_id` válido |
| 429 | Límite de tasa excedido | Espera y reintenta |

| Parámetro | Tipo | Requerido | Descripción |
|---|---|---|---|
| `website_id` | string | Sí | ID del sitio web para el que se verifica el estado de OAuth |

**Respuesta (200):**

```json

```

:::tip
Sondea este endpoint después de que el usuario visite la URL de autorización (Paso 2). Una vez que `oauth_connected` sea `true`, continúa con el Paso 4 para conectar AdSense.

### Paso 4 — Conectar AdSense

**Objetivo:** Activar la conexión de AdSense para el sitio web una vez que OAuth ha sido autorizado.

**Endpoint:** `POST /api/v1/adsense/connect`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/adsense/connect?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Parámetros de consulta:**

| Parámetro | Tipo | Requerido | Descripción |
|---|---|---|---|
| `website_id` | string | Sí | ID del sitio web para el que se conecta AdSense |

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "account_id": "pub-1234567890123456",
    "account_name": "Ejemplo AdSense Demo",
    "publisher_id": "pub-1234567890123456",
    "status": "active",
    "timezone": "America/Guatemala",
    "site_state": "READY",
    "site_auto_ads_enabled": true,
    "connected_at": "2026-04-06T14:35:00Z",
    "error_message": null,
    "last_sync_at": "2026-04-06T14:35:00Z",
    "oauth_connected": true,
    "oauth_email": "publisher@example.com"
  },
  "msg": "AdSense connected successfully"
}
```

:::note
La respuesta es un objeto `AdSenseStatusResponse`. Campos clave:
- `status` — Estado de la conexión (`"active"`, `"disconnected"`, `"error"`)
- `publisher_id` — Tu ID de editor de AdSense (formato: `pub-NNNN`)
- `site_state` — Estado de aprobación de Google para este sitio (p. ej., `"READY"`, `"REQUIRES_REVIEW"`)
- `site_auto_ads_enabled` — Si los anuncios automáticos están habilitados para este sitio
:::

**Errores comunes:**

| Código | Causa | Solución |
|---|---|---|
| 400 | OAuth no autorizado | Completa primero el flujo OAuth (Pasos 2 y 3) |
| 409 | Ya está activo o transición de estado inválida | Verifica el estado actual con `GET /adsense/status`. La conexión puede que ya esté activa |
| 422 | Error de validación — `website_id` ausente o inválido | Proporciona un parámetro de consulta `website_id` válido |

### Paso 11 — Reporte de ingresos personalizado

**Objetivo:** Generar un reporte de ingresos personalizado con rangos de fechas y dimensiones específicas.

**Endpoint:** `POST /api/v1/adsense/earnings/report`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/adsense/earnings/report?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "start_date": "2026-03-01",
    "end_date": "2026-03-31",
    "dimensions": ["DATE", "COUNTRY"]
  }'
```

**Parámetros de consulta:**

| Parámetro | Tipo | Requerido | Descripción |
|---|---|---|---|
| `website_id` | string | Sí | ID del sitio web con una conexión de AdSense activa |

**Cuerpo de la solicitud:**

| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| `start_date` | string | Sí | Fecha de inicio en formato `YYYY-MM-DD` |
| `end_date` | string | Sí | Fecha de fin en formato `YYYY-MM-DD` |
| `dimensions` | array | No | Arreglo de cadenas de dimensión (p. ej., `"DATE"`, `"COUNTRY"`, `"AD_UNIT"`). Por defecto: `["DATE"]` |
| `metrics` | array | No | Arreglo de cadenas de métrica (p. ej., `"ESTIMATED_EARNINGS"`, `"CLICKS"`, `"IMPRESSIONS"`). Por defecto: `["ESTIMATED_EARNINGS", "CLICKS", "IMPRESSIONS"]` |

**Respuesta (200):**

```json
{
  "success": true,
  "data": {
    "headers": [
      { "name": "DATE", "type": "DIMENSION" },
      { "name": "COUNTRY", "type": "DIMENSION" },
      { "name": "ESTIMATED_EARNINGS", "type": "METRIC_CURRENCY", "currency_code": "USD" },
      { "name": "PAGE_VIEWS", "type": "METRIC_TALLY" },
      { "name": "CLICKS", "type": "METRIC_TALLY" }
    ],
    "rows": [
      ["2026-03-01", "US", "12.50", "3200", "85"],
      ["2026-03-01", "MX", "3.25", "1100", "28"],
      ["2026-03-02", "US", "14.00", "3500", "92"],
      ["2026-03-02", "GT", "2.10", "800", "15"]
    ]
  },
  "msg": "Custom report generated successfully"
}
```

:::note
El arreglo `headers` describe cada columna en `rows`. Cada fila es un arreglo de valores de cadena que corresponden a los headers en orden. Los valores de moneda se devuelven como representaciones de cadena de números decimales.
:::

**Errores comunes:**

```bash

BASE_URL="https://api.1platform.pro/api/v1"

# === PASO 0: Obtener el token de aplicación ===

  -H "Content-Type: application/json" \
  -d '{"apiKey": "ak-prod-app-key-2026"}' \
  | jq -r '.data.access_token')

  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "sk-user-publisher-2026"}' \
  | jq -r '.data.access_token')

# === PASO 2: Iniciar la autorización OAuth ===
AUTH_URL=$(curl -s -X GET "$BASE_URL/adsense/oauth/authorize?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq -r '.data.authorization_url')

echo "Abre esta URL en tu navegador: $AUTH_URL"
echo "Presiona Enter después de autorizar..."
read -r

# === PASO 3: Verificar el estado de OAuth ===
OAUTH_STATUS=$(curl -s -X GET "$BASE_URL/adsense/oauth/status?website_id=$WEBSITE_ID" \

  -H "x-user-token: $USER_TOKEN" \
```

## Idempotencia y buenas prácticas

### Idempotencia

:::note
Todos los endpoints `GET` de este flujo son naturalmente idempotentes — pueden llamarse repetidamente sin efectos secundarios. El endpoint `POST /adsense/connect` está protegido por transiciones de estado: llamarlo cuando ya está activo devuelve un conflicto `409`, evitando conexiones duplicadas.
:::

### Reintentos

- Si recibes un **429** (límite de tasa), espera el tiempo indicado y reintenta.
- Si recibes un **500** o un **timeout**, verifica el estado actual con `GET /adsense/status` antes de reintentar una operación de conexión o desconexión.
- Para un **401** (token expirado), regenera el token y reintenta la solicitud original.

### Timeouts

:::warning
Las operaciones de autorización OAuth y de conexión implican llamadas externas a las APIs de Google. Configura tu cliente HTTP con un timeout de al menos **30 segundos** para `POST /adsense/connect` y `POST /adsense/sites/sync`. Los endpoints de reportes de ingresos pueden tardar más en rangos de fechas amplios — usa un timeout de **60 segundos** para `POST /adsense/earnings/report`.
:::

### Tokens

- Ambos tokens (de aplicación y de usuario) expiran en **30 minutos** (1800 segundos). Implementa la renovación automática antes de la expiración.
- Ambos tokens son requeridos en todos los endpoints de AdSense.

### Validaciones importantes

- El sitio web debe estar registrado mediante el flujo de Gestión de Sitios Web antes de conectar AdSense.
- OAuth debe completarse (Pasos 2-3) antes de conectar (Paso 4).
- El parámetro de consulta `website_id` es requerido en cada endpoint de AdSense.
- Desconecta (Paso 14) antes de revocar OAuth (Paso 15) si la conexión está activa.
- El endpoint de reparación (Paso 16) es solo para administradores y no debe llamarse desde flujos de usuarios regulares.
