# Límites de uso

La **Atlas API** aplica límites de uso por cliente para proteger las superficies públicas compartidas. Cuando excedes
un límite recibes un HTTP `429` con el envoltorio de error estándar:

```json
{ "code": "RATE_LIMITED", "message": "Too many requests" }
```

El límite de uso se basa principalmente en la dirección IP del cliente; algunos endpoints propensos a abuso agregan el
user-agent a la clave. Los límites se aplican en el **borde de rutas específicas**, no de manera uniforme a cada
endpoint.

## Qué endpoints están limitados

Los límites se concentran en superficies públicas anónimas de alto volumen. Ejemplos:

| Endpoint | Límite |
|---|---|
| `GET /tenants/by-host` (búsqueda de branding público) | ~60 solicitudes/minuto |
| `GET /public/suggest` (sugerencias de type-ahead) | ~30 solicitudes/minuto |

Otros endpoints pueden tener sus propios límites. Los flujos autenticados y de bajo volumen (inicio de sesión, tickets de
entrega) generalmente no son el cuello de botella — pero maneja siempre `429` de forma defensiva, ya que los límites pueden
ajustarse por entorno.

> Trata los números exactos como detalles operativos que pueden cambiar. Construye tu cliente para que **reaccione a `429`**
> en lugar de codificar umbrales de forma fija.

## Estrategia de backoff

Ante `429`, aplica backoff y reintenta con backoff exponencial más jitter. Reintenta automáticamente solo lecturas idempotentes:

```javascript
async function getWithBackoff(url, init, { maxRetries = 4 } = {}) {
  for (let attempt = 0; ; attempt++) {
    const res = await fetch(url, init);
    if (res.status !== 429) return res;
    if (attempt >= maxRetries) return res; // rendirse; deja que el llamador lo maneje

    // backoff exponencial con jitter: 0.5s, 1s, 2s, 4s (±20%)
    const base = 500 * 2 ** attempt;
    const jitter = base * (Math.random() * 0.4 - 0.2);
    await new Promise((r) => setTimeout(r, base + jitter));
  }
}
```

## Evitar los límites

- **Cachea las lecturas públicas.** El catálogo, el detalle de ítem y las respuestas de branding cambian lentamente — cáchealas en
  tu lado (y deja que tu CDN cachee las superficies de SEO) en lugar de volver a obtenerlas por cada vista de página.
- **Aplica debounce al type-ahead.** Espera una pausa en la escritura antes de llamar a `GET /public/suggest`, y limita
  la tasa de solicitudes por usuario.
- **No hagas polling en bucles ajustados.** Tras el checkout, llama a `entitlements/refresh` una vez (es idempotente)
  en lugar de hacer polling de `entitlements/check` en un bucle ajustado.
- **Reutiliza los tokens.** Mantén el JWT de un miembro durante toda su vida útil y usa el token de refresco; no solicites un
  nuevo magic link por solicitud.
