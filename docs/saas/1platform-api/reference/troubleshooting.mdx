# Troubleshooting

This page is organised by **symptom**, so you can grep for the error you're
seeing rather than guessing at root cause.

## "I get 502 Bad Gateway with HTML when creating a transaction"

**Most likely:** your tenant has no payment gateway configured. Since
2026-04-29 the API returns **`422`** with `data.code: PAYMENT_GATEWAY_NOT_CONFIGURED`
JSON instead of `502 HTML`.

If you still see HTML 502:
1. Hit `GET /health` and confirm the deployment includes the fix
   (`version >= 0.x.x`).
2. Inspect `GET /users/billing` — `data.billing.payment_gateway_id` should be
   non-null.
3. If null, configure a gateway via the operator dashboard and re-create the
   transaction.

## "401 on every call even though my key is valid"

Two common causes:

1. **Environment mismatch.** QA keys do not authenticate against prod and
   vice versa. See [Environments](./environments) — inspect the `Host:`
   header your client sends.
2. **Truncated key.** Newly issued keys carry `ak-` / `sk-` prefixes. Copy
   the full string including the prefix; legacy keys without a prefix still
   work via exact-match lookup.

## "My webhook signature verification keeps failing"

Three pitfalls:

1. **Re-stringifying the body.** 1Platform signs the canonical re-serialisation
   of the body, but you must verify against the **raw bytes** that arrived on
   the wire. Use `request.body` / `php://input` / Express `express.raw()` —
   never `JSON.stringify(JSON.parse(body))`.
2. **Clock skew.** The signed material includes the `X-Webhook-Timestamp`
   header. Reject anything older than 5 minutes. NTP-syncing your servers
   prevents most replay-window false positives.
3. **Comparing the prefix.** Strip the `sha256=` prefix from `X-Webhook-Signature`
   **before** comparing. Don't include it in the computed value.

See [webhook security](../webhooks/security) for verifier code in three languages.

## "I'm hitting 429 in development"

You're probably minting a fresh token on every request. The auth endpoints
are limited to 10 / min. Cache tokens for `expires_in - 60s` and reuse them
across calls — see the
[token caching tip](../flows/payments-and-subscriptions#token-caching-pattern).

## "The webhook arrived but my handler ran twice"

Webhooks are delivered **at-least-once**. Your handler must be idempotent.
Dedupe on `(transaction_id, event)` — see
[receiving notifications → idempotency](../webhooks/receiving-notifications#idempotency-on-the-merchant-side).

## "I rotated the webhook secret and now legitimate webhooks fail verification"

The old secret is invalidated immediately when you call
`POST /webhooks/config/rotate-secret`. In-flight webhooks signed with the old
secret will fail verification at your end.

Mitigation: keep the previous secret in your verifier for **24 hours** (accept
either) before removing it. See
[webhook security → secret rotation](../webhooks/security#secret-rotation).

## "The transaction's `amount` looks 7-8x larger than what I sent"

You are reading the **provider currency** (often GTQ) instead of the merchant
currency (USD). The webhook payload includes both:

- `amount` + `currency` — the upstream provider currency
- `merchant_amount` + `merchant_currency` — what you originally sent

**Always credit your seller using `merchant_amount`.** The `amount` field is
informational only — the exchange rate is opaque and applied at charge time.
