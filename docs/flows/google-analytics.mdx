# Flow: Google Analytics

## Overview

This flow documents how to integrate Google Analytics 4 (GA4) with your websites through the 1Platform API. It covers OAuth authorization, property provisioning, tracking tag installation, metrics retrieval (overview, realtime, per-page), custom dimensions, key events, and server-side Measurement Protocol events.

**Use case:** Connect your users' websites to Google Analytics 4, automatically provision properties with custom dimensions and key events, retrieve traffic metrics, and send server-side conversion events — all through a unified API.

**End result:** A fully provisioned GA4 property with tracking installed, custom dimensions for content AI, key events for conversions, and access to overview, realtime, and per-page metrics.

---

## Prerequisites

| Requirement | Details |
|---|---|
| App API Key | Application key (`ak-...`) provided when registering your app |
| User API Key | User key (`sk-...`) obtained when creating the user |
| Website registered | A website must be registered via `POST /users/websites` first |
| Google account | The user needs a Google account with access to Google Analytics |

### Required headers for all requests

```
Authorization: Bearer $APP_TOKEN
x-user-token: $USER_TOKEN
Content-Type: application/json
```

## Data required to provision analytics

### Setup request

| Field | Type | Required | Description |
|---|---|---|---|
| `website_id` | string | Yes | Website ID to provision analytics for |
| `account_id` | string | Yes | Analytics account ID (format: `accounts/123456789`) |
| `timezone` | string | No | Property timezone (default: `"America/Guatemala"`) |
| `currency` | string | No | ISO 4217 currency code (default: `"GTQ"`) |
| `create_content_dimensions` | boolean | No | Create custom dimensions for content AI (default: `true`) |
| `create_key_events` | boolean | No | Create default key events (default: `true`) |
| `create_mp_secret` | boolean | No | Create Measurement Protocol secret (default: `true`) |

### Measurement Protocol event

| Field | Type | Required | Description |
|---|---|---|---|
| `website_id` | string | Yes | Website ID |
| `event_name` | string | Yes | Server-side event name |
| `params` | object | No | Event parameters (key-value pairs) |

## Step-by-step flow

### Step 0 — Get App Token

**Goal:** Authenticate the application and obtain the app JWT token.

**Endpoint:** `POST /api/v1/auth/token`

**Auth:** None (this is the entry point)

```bash
curl -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "'$APP_API_KEY'"}'
```

**Response (200):**

```json
{
  "success": true,
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 1800
  },
  "msg": "App token generated successfully"
}
```

:::note
Save `data.access_token` as `$APP_TOKEN`. Expires in 30 minutes.
:::

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 401 | Invalid API key or inactive app | Verify that `apiKey` is correct and the app is active |
| 422 | Missing or malformed `apiKey` field | Send `{"apiKey": "..."}` in the body |
| 429 | Rate limit exceeded (10 req/min) | Wait and retry |

### Step 2 — Start OAuth Authorization

**Goal:** Generate an OAuth consent URL so the user can authorize access to their Google Analytics account. The URL uses PKCE for security.

**Endpoint:** `GET /api/v1/analytics/oauth/authorize`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X GET "$BASE_URL/analytics/oauth/authorize?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN"
```

**Response (200):**

```json
{
  "success": true,
  "data": {
    "authorize_url": "https://accounts.google.com/o/oauth2/v2/auth?client_id=..."
  },
  "msg": "Open this URL to authorize analytics"
}
```

:::note
Open `data.authorize_url` in a browser popup. After the user grants permission, the OAuth callback (`GET /analytics/oauth/callback`) handles the code exchange automatically and sends a `postMessage` to the opener window.
:::

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 401 | Authentication required | Regenerate tokens |
| 422 | Missing or invalid `website_id` | Provide a valid website ID |
| 429 | Rate limit exceeded (10 req/min) | Wait and retry |

### Step 4 — Provision Analytics Property

**Goal:** Create a new GA4 property for the website. Provisioning runs in the background — poll `GET /analytics/status` for completion.

**Endpoint:** `POST /api/v1/analytics/setup`

**Auth:** `Authorization: Bearer $APP_TOKEN` + `x-user-token: $USER_TOKEN`

```bash
curl -X POST "$BASE_URL/analytics/setup" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "website_id": "'$WEBSITE_ID'",
    "account_id": "'$ACCOUNT_ID'",
    "timezone": "America/Guatemala",
    "currency": "GTQ",
    "create_content_dimensions": true,
    "create_key_events": true,
    "create_mp_secret": true
  }'
```

**Response (202):**

```json
{
  "success": true,
  "data": {
    "property_id": "properties/123456789",
    "measurement_id": "G-ABCDEF1234",
    "status": "provisioning"
  },
  "msg": "Analytics provisioning started. Poll GET /status for completion."
}
```

:::note
Save `data.property_id` as `$PROPERTY_ID` and `data.measurement_id` for tracking tag installation. The provisioning runs asynchronously — poll `GET /analytics/status` until the status changes to `"active"`.
:::

**Common errors:**

| Code | Cause | Solution |
|---|---|---|
| 401 | Authentication required | Regenerate tokens |
| 409 | Analytics already active for this website | Use the existing property; check `GET /analytics/status` |
| 422 | Validation error — invalid `account_id` format | Must match `accounts/\d+` pattern |
| 429 | Rate limit exceeded or provisioning cooldown (10 req/min) | Wait and retry |

## Full end-to-end example

Complete flow from OAuth authorization to viewing metrics.

```bash
# === VARIABLES ===
BASE_URL="https://api.1platform.pro/api/v1"
WEBSITE_ID="65b2c3d4e5f6a789bcdef012"  # From a previously registered website

# === STEP 0: Get App Token ===
APP_TOKEN=$(curl -s -X POST "$BASE_URL/auth/token" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "ak-prod-app-key-2026"}' \
  | jq -r '.data.access_token')

echo "APP_TOKEN: $APP_TOKEN"

# === STEP 1: Get User Token ===
USER_TOKEN=$(curl -s -X POST "$BASE_URL/users/token" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"apiKey": "sk-user-maria-2026"}' \
  | jq -r '.data.access_token')

echo "USER_TOKEN: $USER_TOKEN"

# === STEP 2: Start OAuth ===
AUTHORIZE_URL=$(curl -s -X GET "$BASE_URL/analytics/oauth/authorize?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq -r '.data.authorize_url')

echo "Open in browser: $AUTHORIZE_URL"
# ... user completes OAuth in browser ...

curl -s -X GET "$BASE_URL/analytics/oauth/status?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .

# === STEP 3: List Accounts ===
ACCOUNT_ID=$(curl -s -X GET "$BASE_URL/analytics/accounts?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \

# === STEP 4: Provision Analytics ===
curl -s -X POST "$BASE_URL/analytics/setup" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "website_id": "'$WEBSITE_ID'",
    "account_id": "'$ACCOUNT_ID'",
    "timezone": "America/Guatemala",
    "currency": "GTQ",
    "create_content_dimensions": true,
    "create_key_events": true,
    "create_mp_secret": true
  }' | jq .

# === STEP 5: Poll Status ===
while true; do
  STATUS=$(curl -s -X GET "$BASE_URL/analytics/status?website_id=$WEBSITE_ID" \
    -H "Authorization: Bearer $APP_TOKEN" \
    -H "x-user-token: $USER_TOKEN" \
    | jq -r '.data.status')

  echo "Analytics status: $STATUS"

  if [ "$STATUS" = "active" ] || [ "$STATUS" = "error" ]; then
    break
  fi

  sleep 3
done

# === STEP 8: Get Site Tag ===
curl -s -X GET "$BASE_URL/analytics/streams/tag?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq -r '.data.tag_html'

# === STEP 11: Get Overview Metrics ===
curl -s -X GET "$BASE_URL/analytics/overview?website_id=$WEBSITE_ID&period=30d&compare=true" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .

# === STEP 12: Get Realtime Metrics ===
curl -s -X GET "$BASE_URL/analytics/realtime?website_id=$WEBSITE_ID" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  | jq .

# === STEP 14: Send Server-Side Event ===
curl -s -X POST "$BASE_URL/analytics/mp-event" \
  -H "Authorization: Bearer $APP_TOKEN" \
  -H "x-user-token: $USER_TOKEN" \
  -H "Content-Type: application/json" \

    "website_id": "'$WEBSITE_ID'",
    "event_name": "article_published",
    "params": {
      "article_id": "507f1f77bcf86cd799439011",
      "category": "technology"
    }
  }' | jq .
```

## Idempotency and best practices

### Idempotency
- Provisioning (`POST /analytics/setup`) returns **409** if analytics is already active — safe to retry.
- Metric retrieval endpoints (`GET`) are naturally idempotent.
- Measurement Protocol events (`POST /analytics/mp-event`) are **not idempotent** — each call sends a new event.

### Retries
- If you receive **429** (rate limit), wait the indicated time and retry.
- If you receive **500** or a **timeout**, verify the analytics status before retrying provisioning.
- For **401** (expired token), regenerate the token and retry the original request.

### Timeouts
- OAuth authorization is user-interactive — no timeout applies to the user flow.
- Property provisioning runs asynchronously and may take **10-30 seconds**. Poll `GET /analytics/status`.
- Metric retrieval endpoints typically respond within **3-10 seconds**.
- Configure your HTTP client with at least **30 seconds** timeout for provisioning and **15 seconds** for metric endpoints.

### Tokens
- Both tokens (app and user) expire in **30 minutes** (1800 seconds). Implement automatic renewal before expiration.
- Both tokens are required on all analytics endpoints.

### Important validations
- The website must be registered in the platform (`POST /users/websites`) before connecting analytics.
- The `account_id` must match the format `accounts/\d+`.
- The `currency` must be a valid ISO 4217 code (3 uppercase letters).
- The `period` parameter must be one of: `7d`, `30d`, `90d`, `12m`.
- Deprovisioning requires `confirm=true` and is irreversible.
- OAuth must be completed before listing accounts or provisioning.
